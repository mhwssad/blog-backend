# 源码结构优化方案

本文档基于当前仓库源码、[项目代码编写规范](./项目代码编写规范.md) 和 [项目结构规范](./项目结构规范.md) 做结构盘点，目标不是做一次性大搬家，而是给后续开发和重构提供一条更稳、更清晰的收口路线。

## 1. 目标与范围

- 目标：让当前项目在继续扩展博客、社区、AI、治理能力时，结构仍然可维护、可拆分、可追踪。
- 范围：`src/main/java/com/cybzacg/blogbackend`、`src/main/resources`、`docs` 入口文档。
- 时间基线：2026-04-28 当前仓库状态。

## 2. 当前总体判断

当前项目的整体方向是对的：

- 已形成 `config / common / core / domain / mapper / module / utils` 的基础分层。
- 主要业务域已经放入 `module` 下，Repository 体系也完成了大部分收口。
- 代码规范、结构规范、任务文档、接口文档都已经建立，说明项目已经具备持续演进的基础。

当前最主要的问题不在“完全没结构”，而在以下三类增量失衡：

- 少量历史入口仍然绕过了既定分层。
- 个别模块已经变大，但还没有完成子域化拆分。
- 顶层 `domain / mapper / utils` 还偏平铺，继续增长后会明显影响可维护性。

一句话判断：

**当前项目已经从“无序开发阶段”进入“需要主动结构治理阶段”，继续加功能没有问题，但如果不做结构收口，后续 chat / file / auth / article 会率先变成高成本模块。**

## 3. 已识别的核心结构问题

### 3.1 Controller 直接依赖 Repository，绕过 Service 边界

状态：`已完成首轮收口（2026-04-28）`

典型位置：

- `src/main/java/com/cybzacg/blogbackend/module/auth/experience/controller/UserExperienceController.java`

当前问题：

- Controller 直接注入 `SysUserRepository`。
- Controller 内直接读取 `SysUser`、计算等级阈值、组装 `UserLevelInfoVO`。

影响：

- 违反“Controller 只接参、鉴权、调 Service、统一返回”的规范。
- 后续等级展示口径一旦变化，控制器会成为业务规则散落点。

建议：

- 新增 `UserExperienceQueryService`，或把当前查询能力合并进现有 `UserExperienceService`。
- Controller 只保留 `userExperienceService.getCurrentUserLevelInfo()` 这一类调用。

当前落实：

- 已将当前等级信息查询收口到 `UserExperienceService#getLevelInfo(Long userId)`。
- `UserExperienceController` 已移除对 `SysUserRepository` 的直接依赖。

优先级：`P0`

### 3.2 Service 层仍有持久化实现泄漏

状态：`已完成首轮收口（2026-04-28）`

典型位置：

- `src/main/java/com/cybzacg/blogbackend/module/auth/service/impl/SysConfigServiceImpl.java`

当前问题：

- `SysConfigServiceImpl` 直接 `extends ServiceImpl<SysConfigMapper, SysConfig>`。
- 直接使用 `baseMapper.selectByConfigKey(...)`。

影响：

- 破坏了“Service 负责业务编排，Repository 负责数据访问组织”的边界。
- 后续任何一个模块都可能继续沿用这种写法，把 Repository 收口重新打散。

建议：

- 保留 `SysConfigService` 作为业务服务。
- 所有配置读取、按 key 查询、基础更新统一下沉到 `SysConfigRepository`。
- `SysConfigServiceImpl` 改为只依赖 `SysConfigRepository + RedisOperator`。

当前落实：

- `SysConfigService` 已移除 `IService<SysConfig>` 继承。
- `SysConfigServiceImpl` 已不再继承 `ServiceImpl`，也不再使用 `baseMapper`。
- 配置更新改为显式 `updateConfig(SysConfig config)` 业务入口，并在服务层统一处理缓存失效。

优先级：`P0`

### 3.3 跨模块直接依赖对方 Repository，模块边界偏硬耦合

状态：`已完成 auth -> article、content -> article、chat -> file 首轮收口（2026-04-28）`

典型位置：

- `src/main/java/com/cybzacg/blogbackend/module/chat/service/impl/UserChatServiceImpl.java`
- `src/main/java/com/cybzacg/blogbackend/module/chat/service/impl/ChatAdminServiceImpl.java`
- `src/main/java/com/cybzacg/blogbackend/module/content/service/impl/UserCommentServiceImpl.java`
- `src/main/java/com/cybzacg/blogbackend/module/content/service/impl/UserCollectionServiceImpl.java`
- `src/main/java/com/cybzacg/blogbackend/module/auth/service/impl/PublicAuthorProfileServiceImpl.java`

当前问题：

- `chat` 直接依赖 `file` Repository。
- `content` 直接依赖 `article` Repository。
- `auth` 直接依赖 `article` Repository 获取作者主页摘要统计。

影响：

- 一个模块会逐渐知道另一个模块的“内部数据拼装方式”，后续改 Repository 口径时影响面很难控。
- 业务能力无法沉淀成稳定服务入口，后面接论坛、AI、通知联动时会继续复制依赖。

建议：

- 逐步建立跨模块 Facade / Query Service：
- `article` 提供 `ArticleProfileQueryService`、`ArticleInteractionGuardService` 这类稳定能力。
- `file` 提供 `FileReferenceService`、`FileAttachmentQueryService`。
- `chat` 内部附件、会话、消息治理先在本模块细分服务，再暴露稳定入口。

当前落实：

- `article` 已新增对外 facade / query service，先收口作者主页统计、文章访问校验和文章计数调整。
- `PublicAuthorProfileServiceImpl` 已不再直接依赖 `BlogArticleRepository` / `BlogArticleSeriesRepository`。
- `UserCommentServiceImpl`、`UserCollectionServiceImpl`、`PublicContentQueryServiceImpl`、`UserFootprintServiceImpl`、`InteractionAdminServiceImpl`、`CommentAdminServiceImpl`、`CollectionAdminServiceImpl` 已改为依赖 `article` facade，而非直接依赖文章 Repository。
- `chat` 已补 `file` facade，`UserChatServiceImpl`、`ChatAdminServiceImpl`、`ChatAttachmentAsyncProcessingServiceImpl` 不再直接依赖文件 Repository / 生命周期服务。

优先级：`P0-P1`

### 3.4 超长 Service 已经进入“高回归风险区”

当前扫描结果：

- `UserChatServiceImpl.java`：约 `1520` 行
- `UserFileServiceImpl.java`：约 `817` 行
- `ChatAdminServiceImpl.java`：约 `762` 行
- `ArticleAdminServiceImpl.java`：约 `531` 行
- `ArticleSeriesServiceImpl.java`：约 `523` 行
- `ChatAttachmentAsyncProcessingServiceImpl.java`：约 `493` 行

影响：

- 单个类承担过多职责，变更很容易互相牵连。
- review、排障、加新规则的成本会持续上升。
- 方法级注释再多，也无法替代职责拆分。

建议：

- `chat` 至少拆成：会话编排、成员治理、消息发送、消息展示组装、附件处理、实时推送编排。
- `file` 至少拆成：上传任务初始化、秒传、分片上传、完成态收口、业务引用维护。
- `article` 至少拆成：文章主体保存、分类标签关系同步、访问授权管理、删除清理、作者配额校验。

优先级：`P1`

### 3.5 `auth` 模块已经具备子域化条件，但当前仍偏平铺

当前情况：

- `module/auth` 约 `152` 个 Java 文件。
- 其中已经出现 `experience`、`token`、`authentication` 子域化迹象。
- 但账号认证、RBAC、作者申请、通知中心、配置、日志、经验体系仍然大量共存于根层级。

影响：

- 后续再加“超级管理员安全”“双重验证”“审计”“账号接管”等能力时，`auth` 会继续膨胀。
- 新同学进入项目时，很难快速建立 `auth` 的业务边界认知。

建议：

- 保留 `auth` 作为一级业务域，但在其内部明确二级子域：
- `account`：登录、注册、当前用户、令牌、认证 provider。
- `rbac`：用户/角色/菜单/授权。
- `author`：作者申请、作者身份、作者主页摘要。
- `notice`：通知中心、通知设置、系统通知投递。
- `experience`：等级、经验、配置、经验流水。
- `config`：系统配置查询与缓存。
- `audit`：后续超级管理员安全、高风险操作审计。

优先级：`P1`

### 3.6 `domain` 与 `mapper` 顶层已出现“平铺过载”信号

当前情况：

- `src/main/java/com/cybzacg/blogbackend/domain` 下约 `47` 个实体文件。
- `src/main/java/com/cybzacg/blogbackend/mapper` 下约 `47` 个 Mapper 文件。
- `src/main/resources/com/cybzacg/blogbackend/mapper` 同样以平铺 XML 为主。

影响：

- 当前还能接受，但继续引入论坛、AI、举报、频道治理后，检索成本会明显上升。
- 同名语义对象增多后，顶层平铺会降低目录辨识度。

建议：

- 中期引入子包分组，但保持顶层职责不变：
- `domain/auth`、`domain/article`、`domain/chat`、`domain/file`、`domain/follow`、`domain/report`、`domain/ai`
- `mapper/auth`、`mapper/article`、`mapper/chat` ...
- `resources/.../mapper/auth/*.xml` 同步分组

优先级：`P2`

### 3.7 `utils` 正在吸收过多“伪通用”能力

当前较大的工具类：

- `FileUtils.java`：约 `789` 行
- `PathUtils.java`：约 `628` 行
- `HttpUtil.java`：约 `606` 行
- `InputStreamUtils.java`：约 `585` 行
- `StringTemplateUtils.java`：约 `411` 行
- `BeanConverterUtils.java`：约 `298` 行

影响：

- 工具类越大，越容易混入模块专属逻辑。
- 真正通用的工具和“只是当前几个地方共用”的工具混在一起，后续会降低复用判断的准确性。

建议：

- 继续保留真正通用的基础工具。
- 将明显偏文件域、存储域、Web 请求域的能力逐步迁往更贴近业务或基础设施的位置：
- 文件 / 路径相关优先进入 `module/file/support` 或 `common/storage/support`
- HTTP 请求工具优先进入 `core/web/support`
- 模板拼接、对象转换类继续评估是否应保留在 `utils`

优先级：`P1-P2`

### 3.8 `ai` 和 `report` 目前还是“结构预留模块”，缺少明确状态定义

当前情况：

- `module/ai` 目前仅有 Repository。
- `module/report` 目前仅有 Repository。

影响：

- 这类“半模块”如果长期存在，后续容易出现两种问题：
- 开发者误以为模块已完成标准分层。
- 后续功能落地时继续随意堆代码，没有统一起点。

建议：

- 在结构文档里增加“预备模块 / schema-aligned module”定义。
- 预备模块阶段允许只有 `repository`，但必须满足：
- 有明确任务文档指向。
- 有 `package-info` 或说明文档标注“当前仅完成表结构对齐，不代表业务模块已落地”。
- 一旦进入接口开发阶段，必须补齐 `service / model / controller / convert` 标准结构。

优先级：`P1`

## 4. 建议的目标结构

## 4.1 顶层保持不变

继续保持：

```text
src/main/java/com/cybzacg/blogbackend
├─ config
├─ common
├─ core
├─ domain
├─ enums
├─ exception
├─ mapper
├─ module
└─ utils
```

当前问题不在顶层目录错误，而在模块内部收口程度不足。

## 4.2 一级模块保持按业务域拆分

继续保持：

- `auth`
- `article`
- `content`
- `file`
- `chat`
- `follow`
- `ai`
- `report`

不建议现在把 `auth`、`content`、`chat` 直接再拆成多个一级模块，否则会把当前改动面放大。

## 4.3 在重型模块内部引入“二级子域”

建议优先对以下模块启用子域化：

### `module/auth`

建议演进为：

```text
module/auth
├─ account
├─ author
├─ rbac
├─ notice
├─ experience
├─ config
├─ audit
└─ shared
```

### `module/chat`

建议演进为：

```text
module/chat
├─ conversation
├─ member
├─ message
├─ attachment
├─ governance
├─ push
├─ websocket
└─ shared
```

### `module/content`

建议演进为：

```text
module/content
├─ comment
├─ collection
├─ taxonomy
├─ interaction
├─ footprint
└─ shared
```

说明：

- 这里的 `shared` 仅限模块内共享模型、常量、组装器，不是新的“杂物箱”。
- 如果后续发现某个子域仍然过大，再在子域内部按标准结构继续拆层。

## 5. 优化落地顺序

### 第一阶段：先修边界，不搬目录

目标：先把明显违反现有规范的点收口，避免继续扩散。

建议先做：

1. 把 `UserExperienceController` 改为只依赖 Service。
2. 把 `SysConfigServiceImpl` 从 `ServiceImpl` 中抽出来，完全改为 `Repository + Service` 模式。
3. 识别跨模块 Repository 直连最重的链路，先补 Facade：
   - `chat -> file`
   - `content -> article`
   - `auth -> article`

### 第二阶段：拆重型类

目标：降低单类认知复杂度和回归风险。

建议先拆：

1. `UserChatServiceImpl`
2. `UserFileServiceImpl`
3. `ChatAdminServiceImpl`
4. `ArticleAdminServiceImpl`

拆分原则：

- 先按职责拆方法和协作类，不先追求目录大搬迁。
- 优先抽“可独立命名的编排能力”，不要只抽几个工具方法。

### 第三阶段：模块内部子域化

目标：让 `auth / chat / content` 的目录结构随规模增长保持稳定。

建议：

- 新增结构规范补充条款，明确“何时允许模块内部按子域拆包”。
- 以后新增功能直接落到子域，不再继续往根层 `service/impl` 平铺。

### 第四阶段：顶层 `domain / mapper / utils` 细分

目标：处理未来论坛、AI、治理模块继续增长后的平铺问题。

建议：

- 只在前面三阶段完成后推进。
- 这是“结构提纯”动作，不是当前最紧急问题。

## 6. 建议新增的结构约束

建议后续把以下规则补入 `项目结构规范.md`：

### 6.1 模块内部子域化触发条件

当满足以下任一条件时，允许在 `module/<domain>` 下按子域继续拆包：

- 模块总文件数超过 `80`
- 单个 `service/impl` 文件超过 `500` 行
- 同一模块已存在 `4` 类以上边界明显不同的业务能力

### 6.2 预备模块定义

新增“预备模块”规则：

- 仅完成表结构和 Repository 对齐的模块，必须显式标注为预备模块。
- 预备模块不允许直接对外暴露接口。
- 进入业务开发后必须补齐标准目录。

### 6.3 跨模块调用规则补充

- 跨模块优先依赖对方 `service` 或 `facade/query service`。
- 非本模块 Repository 只允许在极少数基础设施级场景下使用，并需要在代码评审中单独说明。

## 7. 不建议现在做的事

- 不建议立刻对全仓库做一次性大规模搬包。
- 不建议为了“结构好看”把所有模块都拆成多个一级模块。
- 不建议在没有稳定命名和职责边界前，先机械抽大量 `helper`、`manager`、`processor`。
- 不建议在当前阶段引入复杂代码生成或模块化框架来解决结构问题。

## 8. 结论

当前项目已经有清晰的规范基础，真正需要做的是“把规范从文档继续落回代码”。

建议按下面顺序推进：

1. 先修明显违规边界。
2. 再拆超长 Service。
3. 再做模块内部子域化。
4. 最后处理顶层平铺问题。

如果按这个顺序推进，项目会在不打断当前功能开发节奏的前提下，逐步变得更合理、更健壮。
