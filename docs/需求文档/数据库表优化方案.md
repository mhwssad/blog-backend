# 数据库表优化方案

## 1. 文档目的

本文档用于说明本轮依据需求文档和 PRD 对数据库表结构做的优化方向，帮助后续开发快速理解“为什么要加这些字段和表”。

## 2. 优化原则

- 当前项目处于开发阶段，默认直接修改原始建表脚本。
- 第一阶段只补“明确会落地”的结构，不为远期能力过度设计。
- 先把权限、审核、治理、AI 配置、互动扩展等关键骨架补齐。

## 3. 本轮涉及脚本

- `src/main/resources/mysql/1.sys.sql`
  已直接补齐用户等级、作者申请、通知偏好、AI、举报治理结构。
- `src/main/resources/mysql/02_article.sql`
  已直接补齐文章审核、可见范围、定时发布、系列/专栏结构。
- `src/main/resources/mysql/05_chat.sql`
  已直接补齐频道/群治理、频道申请、入群申请、论坛挂接结构。

## 4. 主要结构优化点

### 4.1 用户与权限侧

新增 / 优化：

- `sys_user.user_level`
- `sys_user.experience_points`
- `sys_user.level_updated_at`
- `sys_author_application`
- `sys_user_notification_setting`

目的：

- 承接用户等级体系
- 承接作者申请与审核
- 承接通知开关配置

### 4.2 博客内容侧

新增 / 优化：

- `blog_article.review_status`
- `blog_article.visibility_scope`
- `blog_article.scheduled_publish_time`
- `blog_article_series`
- `blog_article_series_item`
- `blog_article_review_log`

目的：

- 将文章“发布状态”和“可见范围”从概念上拆得更清楚
- 支撑专栏 / 系列文章
- 支撑后台审核留痕

### 4.3 聊天与频道侧

新增 / 优化：

- `chat_conversation.scene_type`
- `chat_conversation.visibility_scope`
- `chat_conversation.allow_guest_view`
- `chat_conversation.require_join_to_speak`
- `chat_conversation.join_rule`
- `chat_conversation.speak_level_limit`
- `chat_conversation.member_limit`
- `chat_conversation.announcement`
- `chat_conversation.slow_mode_seconds`
- `chat_conversation.display_sort`
- `chat_conversation.channel_category_code`
- `chat_channel_create_application`
- `chat_group_join_application`
- `forum_post_channel_link`

目的：

- 区分单聊、普通群、大厅频道、主题频道等业务场景
- 支撑访客可见、加入后发言、审批入群、等级发言限制等规则
- 为论坛帖子手动挂到频道保留结构

### 4.4 AI 侧

新增：

- `ai_channel_config`
- `ai_chat_session`
- `ai_chat_message`
- `ai_usage_log`

目的：

- 支撑多渠道 / 多模型配置
- 支撑 AI 会话和消息留存
- 支撑额度控制与调用统计

### 4.5 举报治理侧

新增：

- `sys_report_record`
- `sys_report_handle_log`

目的：

- 统一文章、评论、聊天消息举报入口
- 留存状态流转、处理结果和处罚类型
- 为后续申诉和自动治理扩展预留空间

## 5. 初始化与升级路径

空库初始化顺序：

1. `1.sys.sql`
2. `02_article.sql`
3. `04_file.sql`
4. `05_chat.sql`
5. `06_follow.sql`
6. `03_permission_init.sql`

开发阶段执行路径：

- 统一直接维护原始建表脚本
- 只维护当前实际使用的原始建表脚本

## 6. 当前边界

本轮表优化主要解决“结构预留”和“规则落位”问题，尚未表示所有业务逻辑已实现。

尤其以下能力目前只是先把库设计补齐：

- 用户等级具体阈值与经验算法
- 作者申请完整后台流程
- 举报处理完整后台交互
- 论坛正式帖子体系
- AI 知识库 / RAG / agents

## 7. 后续使用建议

- 新开发优先复用本轮新增字段和表，不要再平行创建同义结构。
- 若后续规则调整，优先先改对应原始建表脚本，再补文档。
- 若后续结构调整，继续直接更新对应原始建表脚本。
