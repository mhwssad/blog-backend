# 举报处理流转草案

## 1. 文档信息

- 文档名称：举报处理流转草案
- 所属项目：`blog-backend`
- 文档版本：`v0.1`
- 文档状态：草案
- 整理日期：2026-04-28
- 上级文档：`docs/需求文档/后台治理模块-PRD.md`

## 2. 设计目标

举报处理体系用于支撑平台基础治理能力，目标包括：

- 统一收口文章、评论、聊天消息的举报入口
- 为管理员提供清晰处理流转
- 为处罚动作和留档统计提供标准化结构
- 为后续申诉、自动治理和审核增强预留扩展空间

## 3. 举报对象范围

第一阶段支持举报的对象：

- 文章
- 评论
- 聊天消息

## 4. 推荐状态流转

第一阶段推荐采用以下状态：

- 待处理
- 处理中
- 已处理
- 已驳回

状态说明：

- 待处理：用户已提交举报，尚未被管理员接手
- 处理中：管理员已接手，正在核查或等待补充判断
- 已处理：举报成立或已给出明确处理结果
- 已驳回：举报不成立或证据不足，不做处罚

## 5. 状态操作权限

- 普通管理员：可将待处理转为处理中，可完成已处理或已驳回
- 超级管理员：可接管任意举报单，可直接终结举报流程

推荐补充规则：

- 举报一旦进入已处理或已驳回，普通用户不可再直接修改
- 超级管理员保留特殊修正能力

## 6. 处理结果类型

第一阶段推荐支持以下处理结果：

- 删除内容
- 撤回消息
- 禁言用户
- 封禁账号
- 仅记录不处罚

说明：

- 同一个举报单允许存在“举报成立但仅记录不处罚”的情况
- 处罚动作应与处理结果留档绑定

## 7. 处罚类型建议

为便于留档和统计，第一阶段建议保留处罚类型字段，至少区分：

- 内容删除类
- 消息撤回类
- 临时禁言类
- 账号封禁类
- 无处罚记录类

## 8. 推荐处理流程

推荐流程如下：

1. 用户提交举报
2. 系统生成举报记录，状态为待处理
3. 管理员领取或接手举报，状态改为处理中
4. 管理员核查对象内容、上下文和用户历史行为
5. 管理员给出处理结果
6. 举报流转为已处理或已驳回
7. 系统记录处理人、处理时间、处理结果和处罚类型

## 9. 留档要求

每个举报单建议至少记录：

- 举报对象类型
- 举报对象 ID
- 举报人
- 举报原因
- 举报时间
- 当前状态
- 处理人
- 处理时间
- 处理结果
- 处罚类型
- 备注说明

## 10. 通知建议

第一阶段推荐：

- 举报提交后，至少后台可见
- 被举报对象是否立即通知，第一阶段可谨慎处理
- 举报处理结果是否回告举报人，建议预留但可后续实现

## 11. 申诉规则

第一阶段建议：

- 先不做申诉流程
- 先保证举报处理主链路可用
- 申诉作为后续治理增强能力再补充

## 12. 风险与限制

第一阶段主要风险：

- 恶意举报
- 管理员处理标准不一致
- 处罚动作和记录不同步
- 举报对象已删除导致上下文缺失

## 13. 第一阶段建议结论

第一阶段优先做到：

- 统一举报入口
- 四状态流转
- 管理员 / 超级管理员处理权限区分
- 处罚类型留档
- 处理结果标准化
- 暂不做申诉

## 14. 待继续澄清事项

- 举报原因分类枚举
- 举报结果是否通知举报人
- 是否支持批量处理举报
- 处罚时长和处罚等级体系
