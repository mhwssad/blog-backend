## 2. 文档目标

本文档用于把当前需求文档收敛成一份可直接执行的研发任务清单，目标不是“描述需求”，而是回答以下问题：

- 先做什么
- 每项任务做到什么算完成
- 每项任务依赖什么
- 每项任务默认要补哪些代码、测试和文档

## 3. 执行规则

### 3.1 总规则

- 本清单默认以“当前仓库已有 6 个业务模块能力”为前提，不按从零搭建系统的方式拆解。
- 本清单面向当前后端仓库执行，任务默认按“后端可独立落地”的粒度编排。
- 开发阶段统一直接维护原始建表脚本，不额外拆分兼容迁移脚本。
- 已确认废弃的旧入口、重复说明、无效脚本不继续保留，任务实施过程中如发现同类残留，应顺手收口。

### 3.2 每个任务默认包含

除非任务说明里明确排除，否则每个任务默认包含以下交付：

- 表结构 / 枚举 / 状态定义调整
- Controller 接口
- Service 业务逻辑
- Repository / Mapper 数据访问
- 服务级测试
- 必要的 WebMvc / 权限测试
- `docs/api文档` 中对应接口文档更新
- `docs/需求文档` 中对应需求文档回写

### 3.3 默认完成定义

一个任务只有同时满足以下条件，才可以标记为“已完成”：

- 代码已落地并可运行
- 核心状态流转和异常分支已有测试覆盖
- 前端可见接口文档已更新
- 需求文档已同步回写
- 权限边界与数据落库结果已完成基本验证

## 4. 状态定义

- `待开始`：需求已明确，尚未进入开发
- `设计中`：规则已开始细化，尚未开始主实现
- `进行中`：后端主实现已开始
- `待联调`：后端主实现已完成，等待联调验证
- `待验收`：联调已完成，等待按验收标准确认
- `已完成`：满足当前阶段完成定义
- `后置`：当前不进入开发，先留在待办池

## 7. 任务清单

### T01 作者申请与审核闭环

- 所属模块：`auth` / `article` / 后台治理
- 当前状态：`待开始`
- 优先级：`P0`
- 前置依赖：无
- 来源文档：`项目需求草案-PRD`、`博客模块-PRD`、`作者申请与审核草案`
- 任务目标：
  - 建立普通用户申请作者、管理员审核、作者角色生效的完整闭环
- 研发范围：
  - 用户侧作者申请接口
  - 用户侧申请记录查询接口
  - 管理端审核接口
  - 申请状态流转：待审核 / 已通过 / 已拒绝 / 待补充
  - 作者角色生效后的权限联动
- 开发检查项：
  - 补齐申请参数校验
  - 补齐审核意见和审核人落库
  - 补齐重复申请、重复审核、状态非法流转拦截
  - 补齐角色授予后的权限验证测试
- 验收标准：
  - 注册用户可提交作者申请
  - 管理员可审核并写入审核意见
  - 审核通过后作者增强权限立即生效
  - 审核拒绝和待补充状态可回查

- 任务目标：
  - 建立可落库、可累计、可限额、可影响权限的等级体系
- 研发范围：
  - 经验来源规则：登录活跃、发文章、评论、点赞、被点赞、聊天活跃
  - 等级计算规则
  - 每日经验总上限
  - 聊天经验上限
  - 等级与大厅发言、主题频道发言、建群、申请建频道、AI 次数联动

- 验收标准：
  - 用户行为可正确累计经验值

  - 第一阶段不发生自动降级

- 优先级：`P0`

- 来源文档：`项目需求草案-PRD`、`博客模块-PRD`
- 任务目标：
  - 完整落地文章多状态流转和多可见范围控制
- 研发范围：
  - 文章状态收口：草稿 / 审核中 / 已拒绝 / 已发布 / 定时发布
