## 2. 模块定位

后台治理模块负责对平台用户、内容、聊天、AI 和系统配置进行统一管理，是平台稳定运行、权限控制和异常处置的中枢。

当前第一阶段已完成用户管理、RBAC、内容治理、聊天治理、AI 配置、举报处理、通知偏好、账号封禁、会话级禁言、审计日志和首批数据看板。第二期治理重点转向跨场景禁言、举报结果通知、AI / agent 通知和运营看板扩展。

## 3. 模块目标

- 建立全覆盖后台管理能力。
- 明确普通管理员与超级管理员边界。
- 支撑内容审核、聊天治理、举报处理和系统运营。
- 为后续平台扩展提供统一治理入口。
- 建立全站 / 分场景禁言体系，统一治理聊天、频道、群和后续论坛互动。
- 补齐举报处理结果通知和处罚策略留痕。
- 将 AI / RAG / agent 任务纳入后台运营和通知体系。

## 4. 管理角色

- 普通管理员：负责授权范围内的后台管理。
- 超级管理员：负责高权限处置、管理员管理和异常处理。

## 5. 第一阶段范围

后台治理第一阶段至少包括：

- 用户与权限管理
- 内容审核
- 聊天审核
- 频道与群管理
- 举报处理
- AI 配置与调用统计
- 系统公告
- 数据统计看板

## 5.1 当前已完成基线

当前后台治理已具备：

- 用户、角色、菜单、配置、通知和日志管理。
- 超级管理员二次验证、账号接管、封禁和高风险审计。
- 文章审核、文章状态治理、评论治理。
- 聊天会话、成员、消息治理。
- 举报文章、评论、聊天消息。
- 举报领取、接管、处理、驳回和处理日志。
- 系统通知、用户通知偏好和后台首批数据看板。

当前尚未完成：

- 全站 / 大厅 / 主题频道 / 群等分范围禁言。
- 举报结果通知举报人。
- 被举报人通知策略。
- 举报对象扩展到论坛帖子和论坛回复。
- AI / RAG / agent 任务完成通知。
- 论坛、RAG、agent、举报处理耗时和处罚分布等看板指标。

## 5.2 第二期范围

第二期后台治理包括：

- 禁言体系增强。
- 举报通知增强。
- AI / agent 通知接入。
- 后台看板扩展。
- 论坛治理接入。

## 6. 权限模型

### 6.1 普通管理员

- 由超级管理员创建
- 由超级管理员分配权限
- 仅能在授权范围内执行管理动作

### 6.2 超级管理员

- 固定高权限账号
- 具备必要邮箱
- 是唯一可绕过所有权限和状态限制的账号
- 可修改其他管理员权限
- 可接管任意账号
- 必须启用双重验证
- 所有高风险操作必须留审计记录

## 7. 核心功能需求

### 7.1 用户与权限管理

需支持：

- 用户管理
- 角色管理
- 管理员创建
- 管理员权限分配
- 高权限账号处置

### 7.2 内容治理

需支持：

- 文章审核
- 评论治理
- 内容状态管理

### 7.3 聊天治理

需支持：

- 频道管理
- 群管理
- 消息治理
- 禁言 / 踢出 / 撤回 / 删除

### 7.4 举报处理

需支持：

- 举报文章
- 举报评论
- 举报聊天消息
- 后台人工处理
- 处理记录留档

推荐状态流转：

- 待处理
- 处理中
- 已处理
- 已驳回

推荐处理结果：

- 删除内容
- 撤回消息
- 禁言用户
- 封禁账号
- 仅记录不处罚

补充规则：

- 需要保留处罚类型，便于留档和统计
- 第一阶段先不做申诉
- 第二期需要支持举报处理结果通知举报人
- 第二期需要评估被举报人通知策略，避免泄露处理人和举报人敏感信息
- 第二期论坛帖子和论坛回复需要纳入举报对象范围

### 7.5 AI 治理

需支持：

- 模型与渠道管理
- AI 开关控制
- 用户额度和限额控制
- 数据读取范围配置
- 调用统计

### 7.6 系统运营

需支持：

- 系统公告
- 频道公告管理
- 数据统计看板

### 7.7 禁言体系增强

第二期禁言体系需支持：

- 禁言范围：全站、大厅、主题频道、群。
- 禁言记录：用户、范围、目标 ID、原因、开始时间、结束时间、操作人。
- 发送消息前统一检查禁言状态。
- 管理员创建、解除和查询禁言记录。
- 举报处理时可选择禁言范围和时长。
- 临时禁言到期后自动恢复或发送时动态判断。

### 7.8 通知治理增强

第二期通知治理需支持：

- 举报处理结果通知举报人。
- 举报驳回、举报成立但不处罚、处罚完成分别定义通知模板。
- 通知内容不得泄露处理人、举报人和内部审核细节。
- 投递前遵守用户通知设置。
- AI / RAG / agent 异步任务完成或失败通知。
- 通知跳转到对应举报记录、AI 会话或 agent 任务。

### 7.9 看板扩展

第二期看板需扩展：

- 论坛发帖数、回复数、热门版块。
- RAG 调用数、命中数和无命中数。
- agent 任务数、成功数、失败数。
- 举报处理耗时。
- 处罚类型分布。

## 8. 审计与安全

- 超级管理员必须双重验证
- 高风险操作必须审计
- 接管账号、修改权限、关键治理动作必须可追踪
- 禁言、封禁、删除内容、撤回消息、agent 高风险配置变更必须留审计记录。
- 通知内容需要避免暴露后台操作人和内部处理细节。

## 9. 非目标

第一阶段后台治理模块不重点处理：

- 自动审核
- 复杂推荐治理
- 多租户后台体系

第二期仍不重点处理：

- 完整申诉流程。
- 自动处罚和自动封禁决策。
- 复杂风控评分体系。
- 多租户运营后台。

## 10. 验收标准

- 普通管理员与超级管理员权限边界清晰生效。
- 内容、聊天、AI 和系统运营后台入口具备可用能力。
- 举报处理和审计记录可用。
- 高风险管理动作具备可追踪性。
- 管理员可按范围执行禁言。
- 举报处理结果可按策略通知相关用户。
- AI / agent 异步任务能接入通知中心。
- 后台看板可查看论坛、RAG、agent 和举报治理增强指标。

## 11. 待澄清事项

- 举报状态流转的更细节点与自动通知规则
- 管理员权限粒度的拆分方式
- 超级管理员账号的日常启用策略
- 被举报人通知策略的默认开关
- 不同禁言范围的优先级和叠加规则
- 处罚类型与处理动作的详细映射
