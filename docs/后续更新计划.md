# 后续更新计划

本文档用于指导当前仓库下一阶段的迭代方向。计划口径以模块级推进为主，面向项目内部协作与排期使用。

## 1. 计划基线

- 计划时间：起始于 2026-03-24 之后的下一阶段迭代
- 计划目标：在不盲目扩张模块面的前提下，优先补齐质量、文档、一致性和高风险链路

计划原则：`先保证已有模块可验证、可维护、可联调，再继续扩功能`、`先处理高风险链路，再处理低优先级体验增强`、`接口行为变化必须同步更新前端文档`、`复杂实体构建优先收口到 mapstruct，其次使用 BeanConverterUtils`

- `content` 域的默认收藏夹与收藏记录装配已迁移到 `ContentModelMapper`，实体转换统一继续向更多复杂流程推进。
- 聊天 / WebSocket 方向已完成 chat v1 落地，并补入文件消息、消息治理和后台回执管理；下一步不再是“是否实现”，而是“如何补实时事件、继续扩展更多消息类型和多节点能力”。
- 数据库下一步转向通知双数据源、历史命名不一致和字符集差异等剩余结构问题。
- `file` 主链路测试已继续完成一轮补强，普通上传、秒传待定、整文件与分片 MD5 校验、分片完成前校验、分片合并失败补偿、重复分片覆盖、后台删除清理与后台权限边界已有基础回归路径，并继续纳入终态任务门禁、查询参数校验与去重冲突回收场景；后续继续补更多一致性边界。

### 第三阶段

- 在已有模块稳定的基础上，`P2` 的公共能力整理、文档收敛与 chat v1 基线收口已完成首轮落地，下一步继续处理剩余数据库历史问题与横向基础设施细化，再评估是否继续扩展新业务面或进行更深层次重构。
- chat 后续若继续扩展，优先从编辑/撤回/群治理实时事件、更多消息类型和多节点推送选型入手；如果要把 delivered 从“在线即 delivered”改成 ACK 驱动，需要单独立项推进。
- 将项目维护重心从“补缺功能”转向“稳定迭代和持续交付”。
