# 后续更新计划

本文档用于指导当前仓库下一阶段的迭代方向。计划口径以模块级推进为主，面向项目内部协作与排期使用。

## 1. 计划基线

- 计划时间：起始于 2026-03-24 之后的下一阶段迭代
- 计划目标：在不盲目扩张模块面的前提下，优先补齐质量、文档、一致性和高风险链路

计划原则：`先保证已有模块可验证、可维护、可联调，再继续扩功能`、`先处理高风险链路，再处理低优先级体验增强`、`接口行为变化必须同步更新前端文档`、`复杂实体构建优先收口到 mapstruct，其次使用 BeanConverterUtils`

## 2. 优先级总览

### 2.1 P0

- 在已建立的 `test` profile 基础上，继续扩大测试隔离能力，避免更多测试依赖开发环境配置。
- 收口文件模块关键流程，优先覆盖上传任务状态、秒传复用、分片合并、引用计数与临时资源清理。
- 持续修正文档与代码不一致的问题，特别是 README 与 `docs/api文档`。

### 2.2 P1

- 强化 `article` 与 `content` 的状态流转、访问控制、统计同步、关联关系维护验证。
- 补齐 `auth` 模块关键权限边界、角色菜单分配、通知收件链路的测试与注释。
- 持续将复杂实体构建迁移到 `mapstruct` / `BeanConverterUtils` 规范路径。

### 2.3 P2

- 进一步整理横向公共能力，例如统一转换、公共校验、异常边界说明、测试工具沉淀。
- 逐步淘汰过期或重复说明，减少文档分散。

## 3. 模块级计划

### 3.1 auth 模块

下一阶段重点：

- 继续核对登录、刷新令牌、退出登录、邮箱验证码登录等链路的边界行为。
- 补强用户、角色、菜单、配置、通知、日志相关后台接口的权限验证和关键测试。
- 梳理通知中心和用户通知收件逻辑，确保读状态、投递和查询行为一致。
- 对职责不直观的方法继续补充 Javadoc，降低维护成本。

完成标志：

- 认证主链路具备稳定回归路径。
- RBAC 与通知主线的高风险操作具备基础自动化验证。
- 权限相关接口的行为与文档保持一致。

### 3.2 article 模块

下一阶段重点：

- 继续强化后台文章保存、状态切换、访问控制分配、公开查询之间的一致性。
- 明确文章与分类、标签、互动数据的联动更新边界。
- 补充文章域高成本方法的文档注释，降低跨模块阅读难度。
- 若仍存在手写实体装配较重的流程，按规范优先改用 `mapstruct` 收口。

完成标志：

- 文章主流程具备明确状态流转和访问控制校验。
- 前后台文章行为对齐，减少联调歧义。

### 3.3 content 模块

下一阶段重点：

- 重点核对分类树构建、标签关联、评论状态、收藏夹、互动、足迹等复杂行为链路。
- 对树结构组装、关系重建、计数同步、权限过滤类方法持续补注释。
- 强化公开内容与用户行为之间的可见性、归属和状态校验。
- 继续完善内容域接口文档中的边界说明与异常场景说明。

完成标志：

- 内容辅助域规则一致，联调过程中的歧义显著下降。
- 复杂读写链路具备可追踪、可回归的说明与测试。

### 3.4 file 模块

下一阶段重点：

- 作为最高风险模块持续优先处理。
- 在已补入秒传收口、引用删除服务级测试的基础上，继续核对上传初始化、分片上传、上传完成、秒传命中、后台文件状态调整、业务引用清理等关键环节。
- 明确文件信息、上传任务、业务引用三类数据的事务边界和一致性约束。
- 对容易误解的方法补充文档注释，重点解释状态流转、副作用和清理策略。
- 持续检查是否存在可以收口到 `mapstruct` / `BeanConverterUtils` 的对象转换重复代码。

完成标志：

- 文件主链路在正常、重复提交、失败回退场景下都具备基础验证。
- 关键流程说明和前端接口文档足够支撑联调。

### 3.5 storage / 公共基础能力

下一阶段重点：

- 梳理文件存储相关公共能力与模块边界，避免业务层直接散落存储细节。
- 统一收口公共转换、公共校验和共性异常处理逻辑。
- 对公共工具中阅读成本高的方法补充文档注释。

完成标志：

- 公共层职责更清晰，业务模块重复代码继续下降。

## 4. 横向计划

### 4.1 测试建设

- 保持现有 `test` profile 可用，继续把更多集成测试接入隔离环境。
- 按模块继续补齐高风险回归测试，优先覆盖 `file`、`auth` 权限边界、`article/content` 状态联动。
- 将已有测试从“存在”推进到“能稳定执行、能反映真实风险”。

### 4.2 文档建设

- 每次接口新增、删除、修改，都同步更新 `docs/api文档`。
- 对 README 中已落后于仓库现状的内容做持续修正。
- 将内部进度、计划、规范文档维持在少量但可信的状态，避免重复文档并行失真。

### 4.3 工程质量

- 在后续代码修改中持续执行“优先 `mapstruct`，其次 `BeanConverterUtils`”的转换策略。
- 对不易理解的方法持续补齐 Javadoc，尤其是状态流转、补偿、树组装、关系维护、计数同步相关方法。
- 非平凡改动继续执行至少一次编译验证，在环境允许时补充更贴近业务的测试验证。

## 5. 阶段性目标

### 第一阶段

- 已完成基础 `test` profile 落地，并继续把更多测试接入隔离环境。
- 文件主链路完成一轮重点收口。
- README 与 API 文档的明显滞后问题完成修正。

### 第二阶段

- `auth`、`article`、`content` 三个模块的高风险场景测试和注释继续补齐。
- 实体转换方式逐步统一，减少散落的手写装配逻辑。

### 第三阶段

- 在已有模块稳定的基础上，再评估是否继续扩展新业务面或进行更深层次重构。
- 将项目维护重心从“补缺功能”转向“稳定迭代和持续交付”。
