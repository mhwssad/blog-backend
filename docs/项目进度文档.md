# 项目进度文档

本文档用于记录当前仓库的真实落地进度，面向项目内部协作、排期和后续验收使用。

## 1. 统计口径

- 统计时间：截至 2026-03-24
- 统计范围：`src/main/java/com/cybzacg/blogbackend` 与 `docs`、`src/test`
- 当前定位：单体博客后端，核心业务模块为 `auth`、`article`、`content`、`file`

## 2. 当前整体进度概览

### 2.1 已落地主线

- 认证与鉴权主线已具备可运行骨架，包含登录、注册、邮箱验证码登录、刷新令牌、退出登录。
- 后台 RBAC 主线已落地，包含用户、角色、菜单、配置、通知、日志等管理接口。
- 内容域主线已落地，包含文章、分类、标签、评论、收藏、互动、足迹等核心能力。
- 文件域主线已落地，包含用户文件上传、分片上传、秒传复用、后台文件管理。

### 2.2 当前规模

- 业务模块数：4 个
- 控制器数量：25 个
- `ServiceImpl` 数量：49 个
- 已存在测试文件：12 个
- 已存在前端接口文档：`auth-api.md`、`content-api.md`、`file-api.md`

### 2.3 当前判断

- 项目已经从“只有框架骨架”进入“核心业务接口基本齐备”的阶段。
- 当前主要短板不在模块空缺，而在测试隔离覆盖范围、边界场景验证、文档持续维护和部分能力的收尾一致性。

## 3. 模块进度

### 3.1 auth 模块

当前完成情况：

- 已落地认证接口与认证模型。
- 已落地令牌管理抽象，以及 JWT / Redis 两种令牌管理实现。
- 已落地用户、角色、菜单、配置、通知、日志、用户通知中心相关控制器与服务。
- 已存在 `AuthModelMapper`、`RbacAdminModelMapper` 等映射层，说明模块已开始按转换职责收口。

当前进度判断：

- `auth` 模块已经达到“后台管理与会话链路可联调”的状态。
- 后续重点不是补齐基础 CRUD，而是继续加强权限边界、测试隔离和文档一致性。

### 3.2 article 模块

当前完成情况：

- 已落地后台文章管理、公开文章查询、用户文章行为三个控制器。
- 已具备文章保存、访问控制、分类关联、公开查询、用户行为等服务分层。
- 已存在 `ArticleModelMapper`，说明文章域转换逻辑已开始收口。
- 已存在文章模块测试，表明该模块已进入可验证状态，而非仅任务拆分阶段。

当前进度判断：

- `article` 模块已具备核心接口形态。
- 当前仍需继续补足发布态、访问控制、前台展示口径和互动联动场景的回归验证。

### 3.3 content 模块

当前完成情况：

- 已落地后台分类、标签、评论、收藏、互动、足迹管理控制器。
- 已落地公开分类、标签、评论接口，以及用户评论、收藏、足迹接口。
- 已具备 `CategoryAdminService`、`TagAdminService`、`CommentAdminService`、`CollectionAdminService`、`InteractionAdminService`、`FootprintAdminService` 等核心服务。
- 已存在 `ContentModelMapper`，内容域模型转换已有统一入口。
- 已存在内容模块测试，说明分类权限、内容安全、用户收藏等场景已开始沉淀验证。

当前进度判断：

- `content` 模块已不再是单点功能，而是形成了较完整的互动与内容辅助域。
- 后续要重点处理的是规则一致性、统计口径一致性、复杂行为链路的回归覆盖。

### 3.4 file 模块

当前完成情况：

- 已落地用户文件控制器和后台文件管理控制器。
- 已形成上传任务、文件信息、分片、业务引用、后台管理的服务拆分。
- 从现有服务命名可以确认，上传初始化、上传任务跟踪、分片处理、业务引用维护已经进入正式实现阶段。
- 已存在文件模块测试，现已覆盖部分后台管理、用户上传，以及秒传收口、引用删除等高风险服务链路。

当前进度判断：

- `file` 模块已具备项目级关键支撑能力。
- 后续重点在于保证上传任务状态、秒传复用、引用计数、临时资源清理的全链路一致性。

当前判断：

## 5. 当前存在的缺口与风险

### 5.1 测试隔离已起步，但覆盖范围仍有限

- 当前已补 `test` profile，基础 Spring 上下文测试不再依赖开发环境 MySQL / Redis。
- 但这还只是第一步，更多集成测试和业务链路验证仍需逐步迁移到隔离环境下运行。

### 5.2 文档与代码仍存在持续同步风险

- README 的测试说明已完成一轮修正。
- 但若后续继续只改代码不改文档，前端联调和内部协作成本仍会持续上升。

### 5.3 模块已较完整，但边界验证仍需补强

- `auth`、`article`、`content`、`file` 四个模块都已有较多控制器和服务。
- 模块越完整，越需要针对权限、状态流转、计数同步、关联关系清理等高风险逻辑做回归验证。

### 5.4 文件域属于高一致性风险区域

- 文件模块涉及上传任务、秒传、分片、引用关系、状态切换、临时资源清理。
- 若缺少统一校验和回归测试，最容易出现脏数据、重复引用和资源泄露问题。

## 6. 当前阶段结论

- 截至 2026-03-24，项目核心业务模块已经基本成型，具备继续联调、回归和收口优化的基础。
- 当前阶段应定义为“核心能力已落地，开始进入质量补强与一致性收口阶段”。
- `P0` 的第一项已经启动并完成首轮落地：测试与开发环境的基础解耦已经建立。
- 后续工作的重点不应再是无边界扩张功能面，而应优先继续扩大测试隔离覆盖、处理关键链路验证、同步文档与高风险模块收口。
