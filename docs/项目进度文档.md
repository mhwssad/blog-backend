# 项目进度文档

本文档用于记录当前仓库的真实落地进度，面向项目内部协作、排期和后续验收使用。

## 1. 统计口径

- 统计时间：截至 2026-03-30
- 统计范围：`src/main/java/com/cybzacg/blogbackend` 与 `docs`、`src/test`
- 当前定位：单体博客后端，核心业务模块为 `auth`、`article`、`content`、`file`

## 2. 当前整体进度概览

### 2.1 已落地主线

- 认证与鉴权主线已具备可运行骨架，包含登录、注册、邮箱验证码登录、刷新令牌、退出登录。
- 后台 RBAC 主线已落地，包含用户、角色、菜单、配置、通知、日志等管理接口。
- 内容域主线已落地，包含文章、分类、标签、评论、收藏、互动、足迹等核心能力。
- 文件域主线已落地，包含用户文件上传、分片上传、秒传复用、后台文件管理。
- 已补入聊天域首版建表脚本，为单聊、群聊、全站特殊群聊以及消息回执能力预留数据库基础。
- 已补入粉丝关注关系建表脚本，为关注、取关、粉丝列表、关注列表与互关判断预留数据库基础。
- 已新增数据库表设计修复文档与存量库修复脚本，开始收口实库缺失表、唯一约束、关键索引和日志表引擎不一致问题。

### 2.2 当前规模

- 业务模块数：4 个
- 控制器数量：27 个
- `ServiceImpl` 数量：58 个
- 已存在测试文件：30 个
- 已存在前端接口文档：`auth-api.md`、`content-api.md`、`file-api.md`、`chat-api.md`

### 2.3 当前判断

- 项目已经从“只有框架骨架”进入“核心业务接口基本齐备”的阶段。
- 当前主要短板不在模块空缺，而在测试隔离覆盖范围、边界场景验证、文档持续维护和部分能力的收尾一致性。
- 聊天 / WebSocket 方向已完成 chat v1 首版实现，包含单聊 / 群聊 / 全站群聊、文本消息收发、会话列表 / 历史消息、已读游标、群管理、后台聊天管理，以及单机版 WebSocket 实时推送。
- 粉丝关注方向也已完成数据库层预留，以及 `domain` / `mapper` / `service` / `xml` 基础骨架，但用户资料统计、关注流与通知联动尚未进入实现阶段。

## 5. 当前存在的缺口与风险

### 5.1 测试隔离已起步，但覆盖范围仍有限

- 当前已补 `test` profile，基础 Spring 上下文测试不再依赖开发环境 MySQL / Redis。
- `file` 相关控制器权限测试也已纳入隔离执行范围，但这还只是阶段性进展，更多集成测试和业务链路验证仍需逐步迁移到隔离环境下运行。

### 5.2 文档与代码仍存在持续同步风险

- README 与 `docs/README.md`、`docs/api文档/README.md` 已完成一轮入口收口，`HELP.md` 过期模板也已移除。
- 但若后续继续只改代码不改文档，前端联调和内部协作成本仍会持续上升。

### 5.3 模块已较完整，但边界验证仍需补强

- `auth`、`article`、`content`、`file` 四个模块都已有较多控制器和服务。
- 模块越完整，越需要针对权限、状态流转、计数同步、关联关系清理等高风险逻辑做回归验证。
- `auth` 模块的通知投递、收件箱、认证主链路、注册基础边界、邮箱验证码关键异常场景，以及角色管理、用户管理、菜单管理、配置管理、日志管理等后台接口关键权限控制已完成首轮测试补强，其中角色/用户主线已完成首轮权限闭环，但更多管理侧异常边界仍需继续覆盖。
- `article` 模块的后台状态切换、指定用户授权拦截与授权重建，以及删除清理链路已完成首轮测试补强，但公开查询口径、更多保存分支与跨模块联动仍需继续覆盖。
- `content` 模块的评论创建/删除、收藏默认夹创建和计数回退已完成首轮测试补强，但更广的互动、足迹和管理侧边界仍需继续覆盖。
- `file` 模块的普通上传、秒传待定返回、MD5 校验失败、分片完成前校验、分片合并失败补偿、重复分片覆盖、后台删除清理与后台权限边界等主链路已完成一轮测试补强，并继续补入终态任务门禁、查询参数校验与去重冲突回收等场景。
- 后续仍需继续扩大 file 模块对更多并发、已删除文件复活、副作用清理与后台管理边界的隔离回归覆盖。

### 5.4 聊天方向已进入业务落地阶段，但测试与扩展边界仍需补强

- 当前已完成聊天域表结构、用户侧收发链路、群管理、后台聊天管理接口，以及单机版 WebSocket 实时推送；聊天主线已经不是“是否实现”的问题，而是“如何持续验证和扩展”。
- 当前 delivered 语义已明确为“在线即 delivered”，对应代码、测试和文档已对齐；如果未来要调整为 ACK 驱动，需要同步改造 recipient、cursor、push 与协议回归，不属于无风险的小改动。
- 当前更大的风险已转向编辑/撤回/群治理缺少专用实时事件、更多消息类型尚未扩展，以及多节点推送等下一阶段基础设施问题。

### 5.5 粉丝关注方向仍停留在数据库准备阶段

- 当前已完成 `sys_user_follow` 建表脚本预留，并补入关注关系基础仓储与服务骨架；但尚未落地用户关注接口、粉丝统计、取关恢复与关注通知联动。
- 现有 sys_interaction 并不适合直接承载关注关系，后续应基于独立关系表推进实现。

### 5.6 数据库历史脚本与实库之间仍有继续对齐空间

- 本轮已开始通过 `07_schema_repair.sql` 收口实库缺失表、唯一约束和索引问题，并继续补入 `sys_role` 软删除兼容唯一索引修复。
- 但通知目标用户字段的双数据源设计、历史时间字段命名不一致、字符集/排序规则不统一等问题仍未完全处理。

### 5.7 文件域属于高一致性风险区域

- 文件模块涉及上传任务、秒传、分片、引用关系、状态切换、临时资源清理。
- 当前已补齐完成上传阶段的一处临时资源收口缺口，并继续收口终态任务门禁、查询参数校验、去重冲突回收和本地分片缺失即失败等高风险边界。
- 但文件域仍然属于高一致性区域，后续仍需持续扩大并发、异常补偿和后台治理场景的回归验证。
