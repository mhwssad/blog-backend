# 项目代码编写规范

本文档只约束“代码应该怎么写、怎么验证、怎么提交”。目录分层、模块组织、模型落位和新增文件放置规则不在本文档展开，统一以 [项目结构规范](./项目结构规范.md) 为准。

## 1. 适用边界

本文档关注：

- Controller、Service、Repository、Mapper 等角色在代码实现中的行为边界。
- 业务流程、事务、一致性、异常、日志、缓存、测试、接口文档和 Git 提交。
- 已有代码的增量修改方式和验证要求。

本文档不关注：

- 顶层包目录怎么划分。
- 新增类应该放在哪个目录。
- 模块是否需要子域化。
- Request、VO、Repository、Convert 等文件的目录落位。

以上结构问题统一查看 [项目结构规范](./项目结构规范.md)。

## 2. 分层编码职责

### 2.1 Controller 编码要求

- Controller 保持轻量，只做参数接收、参数校验注解、鉴权注解、调用 Service 和统一响应包装。
- Controller 不编写业务编排、权限分支、状态流转、计数更新、关联关系维护逻辑。
- Controller 不直接操作 Mapper、Repository 或数据库。
- Controller 返回统一响应包装，例如 `Result`、`PageResult`。
- 参数校验优先使用 Bean Validation 注解或项目自定义校验注解，不在 Controller 中手写校验分支。

### 2.2 Service 编码要求

- Service 负责业务编排、权限校验、状态流转、事务边界、关系维护和计数同步。
- 修改业务数据前必须先校验归属、可见性、权限和状态合法性。
- 多表联动、状态流转、计数更新必须收口在统一事务边界内。
- 同一业务动作涉及多步联动时，方法应保持流程清晰，避免把关键业务拆散到难以追踪。
- Service 不绕过已收口的 Repository 查询条件重新拼装数据访问逻辑。
- Service 不手动捕获业务异常后重新包装成另一类业务异常。

### 2.3 Repository / Mapper 编码要求

- Repository 负责数据访问组织，不承担业务决策。
- Repository 只回答“如何查、如何写”，不回答“为什么允许这么做”。
- Repository 不调用 Service。
- Mapper 和 XML 只承载数据访问，不写业务状态流转和权限判断。
- SQL 或查询条件若具有业务含义，对应业务校验仍应放在 Service。

### 2.4 公共能力编码要求

- 通用能力必须确实跨模块复用，不因“暂时方便”抽成公共工具。
- 通用工具不依赖具体业务模块状态。
- 禁止在公共能力中隐藏业务特例分支。

## 3. 实体转换规范

- 请求对象转实体、实体转 VO、更新已有实体、构建关联记录时，优先使用 MapStruct。
- 仅在 MapStruct 明显不适合时使用 `BeanConverterUtils`，例如简单浅拷贝后补少量运行时字段。
- 能通过 MapStruct 或 `BeanConverterUtils` 表达的转换，不手写大段 `new Entity()` + `setXxx(...)`。
- 转换逻辑不得散落在 Service 主流程中。
- 映射方法应表达业务语义，避免出现难以理解的万能转换方法。

## 4. 业务编写要求

### 4.1 通用要求

- 新增、修改、删除业务数据前，先完成归属、权限、可见性和状态合法性校验。
- 删除业务数据时，同步清理依赖关系、计数、引用和临时资源。
- 对认证、文章发布、通知投递、上传任务、聊天消息等有状态流程，使用显式状态校验和状态流转。
- 除非业务明确要求，否则不要因为外部存储清理失败而阻塞本地元数据收口。
- 对前端可见行为保持稳定；行为变化必须同步接口文档。

### 4.2 高一致性模块要求

文件域：

- 上传任务、分片记录、文件元数据、业务引用和物理文件状态必须保持一致。
- 秒传、分片合并、上传失败补偿和引用归零回收必须有明确状态收口。

聊天域：

- 消息发送、编辑、撤回、删除涉及的消息状态、回执和游标必须保持一致。
- 聊天附件引用必须与文件生命周期联动。
- 会话成员、群治理、禁言、踢出等动作必须同步影响发送权限和相关事件。

治理与安全域：

- 高风险操作必须完成权限校验、必要的二次验证和审计留档。
- 举报处理、封禁、禁言、内容隐藏等治理动作必须调用对应业务服务，不直接绕过一致性逻辑改表。

### 4.3 数据库脚本修改要求

- 当前项目处于开发阶段，数据库结构调整默认直接修改原始建表脚本。
- 新增字段、删除字段、调整字段、增删索引、增删表时，不额外拆兼容迁移脚本，除非用户明确要求。
- 修改表结构时，同步检查实体、Mapper XML、枚举、初始化数据和接口文档。

## 5. 注释与可读性规范

### 5.1 类级注释

- 对职责不直观的服务、控制器、抽象类、接口、策略类补充类级注释。
- 类级注释说明职责边界，不写空泛描述。

### 5.2 方法级注释

以下方法需要补充方法级 Javadoc：

- 业务编排方法。
- 状态流转方法。
- 失败回退或补偿方法。
- 树结构组装方法。
- 访问控制与权限过滤方法。
- 解析逻辑方法。
- 关联关系重建方法。
- 计数同步方法。
- 仅看签名无法直接理解意图的私有 helper。

### 5.3 注释风格

- 注释重点写职责、关键约束和重要副作用。
- 不逐行复述显而易见的赋值语句。
- 行内注释只在代码本身不够自解释时使用。

## 6. 异常处理规范

### 6.1 异常体系

- 统一使用 `BusinessException` + `ResultCode` / `ResultErrorCode` 表达业务异常。
- Service 优先通过 `ExceptionThrowerCore` 抛出业务异常。
- 仅在异常语义稳定且确有必要时新增专用异常子类。

### 6.2 异常信息

- 异常信息面向调用方，提供可理解的描述。
- 不暴露密码、令牌、SQL、路径、内部堆栈等实现细节。
- 不用异常做普通流程控制。

### 6.3 异常处理边界

- 全局异常处理器负责统一响应转换。
- Service 捕获异常只用于补偿、回退或补充上下文。
- 补偿完成后应抛出原始异常或更具语义的异常，不吞异常。

## 7. 日志规范

### 7.1 使用方式

- 使用 Lombok `@Slf4j`，不手动声明 Logger。
- 异步处理、定时任务、WebSocket 推送、外部服务调用必须记录关键节点。

### 7.2 日志级别

- `debug`：开发调试信息。
- `info`：关键业务节点和任务执行结果。
- `warn`：业务降级、补偿失败、外部调用失败但主流程可继续。
- `error`：未预期异常或需要人工介入的错误。

### 7.3 日志内容

- 不记录密码、验证码、令牌原文、完整手机号、完整邮箱敏感部分。
- 正常 CRUD 不添加冗余日志。
- 关键日志应包含必要业务 ID，便于定位。

## 8. 缓存与 Redis 使用规范

- 统一通过 `RedisOperator` 操作 Redis，不直接注入 `RedisTemplate`。
- Key 使用 `domain:sub-function:qualifier:identifier` 层级格式。
- Key 运行时拼接统一使用 `RedisKeyUtils.build()`。
- 禁止在 Service 中硬编码 Redis Key 字符串。
- 所有缓存 Key 必须设置 TTL，不允许永久缓存 Key。
- 写操作优先更新数据库，再更新或失效缓存。
- 读取允许短暂不一致时，必须在业务上可接受。

## 9. 测试编写规范

### 9.1 测试框架

- 使用 JUnit 5 + Mockito。
- 单元测试使用 `@ExtendWith(MockitoExtension.class)`。
- 测试类命名为 `{被测类名}Test`。
- 测试方法使用描述性 CamelCase，例如 `loginShouldRejectLockedAccount`。

### 9.2 Mock 边界

- Service 测试 mock Repository 和外部 Service。
- 不 mock 被测服务自身的内部方法。
- 静态方法 mock 使用 `try-with-resources` 包裹 `MockedStatic`。
- 不过度 mock，以免测试失去验证价值。

### 9.3 覆盖要求

- 新增或修改 Service 方法时，必须补充对应服务级测试。
- 测试覆盖正常路径、边界条件和异常分支。
- 权限、归属、状态流转、计数、事务联动属于高风险路径，必须优先覆盖。
- 若用户明确要求暂缓测试，必须在对应任务文档中标明测试补强项。

## 10. 接口文档规范

- 面向前端的接口文档位于 `docs/api文档`。
- 新增、删除、修改接口时，同步更新对应 API 文档。
- 接口文档必须写清楚请求方式、请求路径、鉴权要求、请求字段、响应字段、枚举含义、异常场景和关键边界。
- 前端可见行为变化不得只改代码不改文档。

## 11. 项目进度与任务文档规范

- `docs/tasks/README.md` 和 `docs/tasks` 下任务清单用于记录当前阶段、任务、优先级和验收标准。
- 新增、调整或完成具体任务时，必须同步更新对应任务清单。
- 影响阶段判断、已完成能力、缺口风险或后续路线时，必须更新任务导航。
- 文档更新必须反映真实状态，不把尚未完成的工作写成已完成。

## 12. 代码修改与验证

- 优先做增量式修改，保持既有风格。
- 不把无关重构、格式清理、行为修改混在一次任务里。
- 已确认废弃的兼容逻辑、旧入口、失效说明和重复文档应及时删除或收口。
- 非平凡代码改动必须在环境允许时执行 `mvn -q -DskipTests compile`。
- 涉及核心业务、权限或状态流转时，应按影响范围补充测试。

## 13. Git 提交规范

### 13.1 提交信息

提交信息使用清晰、可检索的格式：

```text
type(scope): 简要说明
```

示例：

- `feat(file): 新增分片上传完成接口`
- `fix(auth): 修复刷新令牌校验失败问题`
- `refactor(article): 收敛文章保存流程`
- `docs(api): 更新文件接口文档`

### 13.2 type

- `feat`：新增功能。
- `fix`：缺陷修复。
- `refactor`：重构，行为不变。
- `docs`：文档更新。
- `test`：测试补充或调整。
- `chore`：杂项维护。
- `perf`：性能优化。

### 13.3 提交内容

- 一次提交只解决一类问题。
- 提交标题聚焦本次改动，不写“修改代码”“更新一下”。
- 接口行为变化应在提交说明中体现。
- 提交前确认可编译，必要时补充测试或说明未验证原因。

## 14. 什么时候提交 Git

完成一个独立、可验证、可 review 的最小改动单元后再提交。

应单独提交的场景：

- 新增功能完成并通过基本验证。
- 修复 bug 并确认影响范围。
- 完成一次独立重构且行为未变。
- 完成接口文档、README、AGENTS 或规范文档更新。

不应提交的状态：

- 代码明显未完成。
- 项目无法编译。
- 含有临时调试代码。
- 混入与当前目标无关的改动。
