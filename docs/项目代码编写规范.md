# 项目代码编写规范

本文档用于约束当前仓库的日常代码编写、重构、文档维护和 Git 提交行为。

目录分层、模块组织、模型落位与新增代码放置规则，另见 [项目结构规范](./项目结构规范.md)。

## 1. 项目背景

- 本仓库是一个基于 Java 17、Spring Boot 4、Spring Security、MyBatis-Plus、MySQL、Redis、Knife4j/OpenAPI、MapStruct、Lombok
  的博客后端项目。
- 当前核心业务模块包括 `auth`、`article`、`content`、`file`、`chat`、`follow`。
- 代码主目录按 `config`、`core`、`common`、`exception`、`domain`、`mapper`、`module`、`utils` 分层组织。

## 2. 分层与职责

### 2.1 Controller 层

- Controller 保持轻量，只负责参数接收、鉴权注解、调用 Service、返回统一响应。
- 不要在 Controller 中写复杂业务编排、权限分支、状态流转、计数更新、关联关系维护逻辑。
- Controller 返回统一响应包装，例如 `Result`、`PageResult`。

### 2.2 Service 层

- Service 层负责业务编排、权限校验、状态流转、事务边界、关系维护、计数同步。
- 对会影响多表数据、状态字段、统计字段的逻辑，优先在 Service 层完成统一收口。
- 同一业务动作涉及多步联动时，优先设计成可读性明确的方法，不要把关键流程拆得过碎导致理解困难。

### 2.3 Mapper / 持久化层

- 持久化查询和更新逻辑放在 MyBatis-Plus Service / Mapper 层。
- 不要把复杂业务规则塞进 Mapper 或 SQL 拼装里。
- SQL 或查询条件若牵涉业务含义，相关校验仍应放在 Service 层。

### 2.4 Repository 层

- Repository 负责"如何查、如何写"的数据访问组织，不负责"为什么这么做"的业务决策。
- Repository 接口继承 `IService<Entity>`，实现类继承 `ServiceImpl<Mapper, Entity>`。
- 简单查询在实现类中使用 `LambdaQueryWrapper` 内联构建；复杂或多表查询委托给 `baseMapper` 自定义方法。
- 禁止 Repository 调用 Service（反向依赖）。
- 禁止在 Repository 中编写业务校验、状态流转、权限判断逻辑。
- 已通过 Repository 收口的查询条件，禁止在 Service 中绕过 Repository 重新拼装。

### 2.5 公共层

- 横切能力优先放在 `common`、`core`、`config`、`utils` 等公共层。
- 不要在业务模块中重复实现通用能力，例如权限判断、统一异常、序列化、存储路由、公共转换逻辑。

## 3. 实体转换规范

- 优先使用映射方式简化实体构建。
- 请求对象转实体、实体转 VO、更新已有实体、构建关联记录时，优先使用 `mapstruct`。
- 只有在 `mapstruct` 不适合的场景下，才使用 `BeanConverterUtils`，例如简单浅拷贝后再补少量运行时字段。
- 能通过 `mapstruct` 或 `BeanConverterUtils` 表达的转换，不要再手写大段 `new Entity()` + `setXxx(...)`。
- 新增映射逻辑时，优先放在对应业务模块的 convert / mapper 语义位置，避免转换代码散落在 Service 中。

## 4. 业务编写要求

### 4.1 通用要求

- 修改业务数据前，先校验归属、可见性和权限约束。
- 对新增、修改、删除流程中会影响计数、关联表、状态字段的逻辑，保持在同一个事务边界内完成联动更新。
- 对认证、文章发布、通知投递、上传任务、文件生命周期这类有状态流程，使用显式校验和明确状态流转，不要依赖隐式副作用。
- 删除业务数据时，要同步清理依赖关系、计数和临时资源。
- 除非业务明确要求，否则不要因为外部存储清理失败而阻塞本地元数据收口。

### 4.2 高一致性模块要求

涉及有状态流程的模块（如文件上传、聊天消息），必须保证任务状态、业务状态、关联引用之间的数据一致性。

文件域要求：

- 文件域保持现有模式，包括上传任务跟踪、秒传复用、引用计数维护、临时分片清理。
- 上传、分片合并、秒传命中、引用回收等流程必须保证任务状态、文件状态、业务引用之间的数据一致性。

聊天域要求：

- 消息发送、撤回、编辑涉及的消息状态、已读游标、回执记录必须保持一致。
- 聊天附件引用必须与文件模块的业务引用联动，消息删除时附件引用应同步释放。
- 会话成员资格、群治理状态变更时，必须同步推进相关游标、权限和未读计数。

### 4.3 接口契约要求

- 新增或修改接口时，保证 Controller 路径、鉴权规则、请求模型、响应模型、Service 行为保持一致。
- 前端可见的接口契约应保持稳定；如果行为发生变化，必须同步更新接口文档。

### 4.4 数据库脚本修改要求

- 当前项目处于开发阶段，默认不以“兼容旧库/平滑迁移”作为数据库结构调整前提。
- 新增字段、删除字段、调整字段、增删索引、增删表时，优先直接修改原始建表脚本，而不是额外追加兼容性扩展脚本。
- 除非用户明确要求，否则不要为当前开发任务额外拆分独立迁移脚本。
- 涉及模块结构扩展时，优先同步修改对应初始化脚本，例如 `1.sys.sql`、`02_article.sql`、`04_file.sql`、`05_chat.sql`、`06_follow.sql`。

## 5. 注释与可读性规范

### 5.1 类级注释

- 对职责不够直观的模块、服务、控制器、抽象类或接口，补充类级注释。
- 类级注释重点说明职责边界，不要写成空泛描述。

### 5.2 方法级注释

- 对不易理解的方法补充方法级 Javadoc。
- 不易理解的方法包括但不限于：
- 业务编排方法。
- 状态流转方法。
- 失败回退或补偿方法。
- 树结构组装方法。
- 访问控制与权限过滤方法。
- 解析逻辑方法。
- 关联关系重建方法。
- 计数同步方法。
- 仅看签名无法直接理解意图的私有辅助方法。
- 即使是私有 helper，只要阅读成本高，也不能省略方法文档注释。

### 5.3 注释风格

- 方法文档注释保持简洁，重点写职责、关键约束和重要副作用。
- 注释要有信息量，不要逐行复述显而易见的赋值语句。
- 只有在代码本身不够自解释时，才补充少量行内注释。

## 6. 异常处理规范

### 6.1 异常体系

- 统一使用 `BusinessException` + `ResultCode` / `ResultErrorCode` 表达业务异常。
- Service 层优先通过 `ExceptionThrowerCore` 工具类抛异常（如 `throwBusinessIf`、`throwBusinessIfNull`），避免到处
  `new BusinessException()`。
- 仅在异常语义非常稳定且确有必要时才新增专用异常子类，参考 `StorageException`。

### 6.2 异常信息

- 异常信息应面向调用方（前端或上层服务）提供可理解的描述，不暴露内部实现细节。
- 不要用异常做流程控制，只在真正的异常场景下抛出。

### 6.3 异常处理边界

- 全局异常处理器已按优先级分层（Security > MVC > Business > Validation > Database > System）。
- Service 层不要手动捕获业务异常后重新包装为另一种异常类型，避免干扰全局异常处理器的正常分发。
- 如果 Service 层需要捕获异常做补偿或回退，应在处理完成后抛出原始异常或更具语义的异常，不要吞掉异常。

## 7. 日志规范

### 7.1 日志使用

- 使用 Lombok `@Slf4j` 注解，不手动声明 Logger。
- 异步处理、定时任务、WebSocket 推送等基础设施类服务必须记录关键节点日志。

### 7.2 日志级别

- `log.warn`：业务降级、补偿失败、外部调用失败但不影响主流程的场景。
- `log.error`：未预期异常、需要人工介入的错误。
- `log.info`：关键业务节点，如认证成功或失败、文件清理、定时任务执行结果。
- `log.debug`：开发调试用，生产环境不输出。

### 7.3 日志内容

- 日志内容不包含敏感信息（密码、令牌原文、手机号完整号码）。
- 不要在正常 CRUD 流程中添加冗余日志。

## 8. 缓存与 Redis 使用规范

### 8.1 操作入口

- 统一通过 `RedisOperator` 操作 Redis，不直接注入 `RedisTemplate`。

### 8.2 Key 命名

- Key 命名遵循 `domain:sub-function:qualifier:identifier` 层级格式，使用 `:` 分隔。
- Key 前缀定义为常量，放在模块 `constant` 或 `common/constant/RedisConstants` 中。
- Key 运行时拼接统一使用 `RedisKeyUtils.build()`。
- 禁止在 Service 中硬编码 Redis key 字符串。

### 8.3 TTL 策略

- 所有缓存 Key 必须设置 TTL，不允许永久 Key。
- 短时验证码、限流计数使用固定 `Duration`。
- 会话令牌、配置缓存通过配置文件或 `SysConfig` 驱动 TTL。

### 8.4 缓存一致性

- 写操作优先更新数据库再操作缓存。
- 读取时允许短暂不一致，但需在业务上可接受。

## 9. 测试编写规范

### 9.1 测试框架与结构

- 测试框架使用 JUnit 5 + Mockito，测试类标注 `@ExtendWith(MockitoExtension.class)`。
- 测试类命名：`{被测类名}Test`，与生产代码模块路径保持同域对齐。
- 测试方法命名使用描述性 CamelCase，如 `loginShouldRejectLockedAccount`。
- 测试结构：`@Mock` 字段声明所有依赖 + `@BeforeEach` 中通过构造器注入创建被测服务 + `@Test` 方法。

### 9.2 Mock 边界

- Service 测试 mock Repository 和外部 Service，不 mock 被测服务自身的内部方法。
- 静态方法 mock 使用 `try-with-resources` 包裹 `MockedStatic`，如 `SecurityTestUtils.mockUserId()`。
- 不要过度 mock 以致测试失去验证价值。

### 9.3 覆盖要求

- 新增或修改 Service 方法时，必须补充对应的服务级测试。
- 测试应覆盖正常路径、边界条件和异常分支，不要只测 happy path。
- 当前处于接口快速迭代的开发阶段时，如用户明确要求先不补测试，可暂缓新增测试；但需要在对应 `docs/tasks` 文档中标明“测试待接口稳定后统一补齐”。

## 10. 接口文档规范

- 面向前端的接口文档位于 `docs/api文档`。
- 新增、删除、修改接口时，要同步更新对应文档，必要时更新 `docs/api文档/README.md`。
- 接口文档面向前端使用，必须写清楚：
- 请求方式。
- 请求路径。
- 鉴权要求。
- 请求字段。
- 响应字段。
- 枚举或状态含义。
- 关键边界情况和异常场景。
- 不要留下前端可见但未记录的行为变化。

## 11. 项目进度与任务文档规范

- `docs/tasks/README.md` 及 `docs/tasks` 下的任务清单用于记录当前项目阶段、模块进度、后续任务、优先级、执行项和验收标准。
- 完成会影响以下信息的任务后，必须同步更新 `docs/tasks/README.md` 或对应任务清单：
- 当前已完成能力发生变化。
- 模块阶段状态发生变化。
- 当前缺口或风险判断发生变化。
- 阶段目标发生变化。
- 新增、调整或完成具体任务时，必须同步更新 `docs/tasks` 下对应任务清单。
- 文档更新应反映真实状态，不要机械复制提交记录，也不要把尚未完成的工作写成已完成。
- 若本次任务只涉及局部代码但不会影响项目级进度和任务状态，可以不更新任务文档；若会影响，就不能省略。

## 12. 代码修改预期

- 优先做增量式修改，尽量保持各模块既有风格和约定。
- 修改旧代码时，如果附近代码的可读性问题会直接影响本次改动理解，可以顺手做小范围整理。
- 不要把无关重构、格式清理、行为修改混在一次任务里，除非它们强相关且必须同时完成。
- 当前项目处于开发阶段时，代码、脚本和文档应主动保持清爽；已经确认废弃的兼容逻辑、旧入口、失效说明和重复文档应及时删除或收口，不继续长期保留。
- 新增说明文档、脚本入口或结构约定前，先判断是否能并入现有主入口，避免在开发阶段堆积多套并行说明和历史残留。
- 对非平凡代码改动，除非环境阻塞，否则至少执行一次 `mvn -q -DskipTests compile` 进行验证。

## 13. Git 提交规范

### 13.1 提交信息格式

- 提交信息使用清晰、可检索的中文或中英混合风格。
- 推荐格式：`type(scope): 简要说明`

示例：

- `feat(file): 新增分片上传完成接口`
- `fix(auth): 修复刷新令牌校验失败问题`
- `refactor(article): 收敛文章保存流程的分类标签同步逻辑`
- `docs(api): 更新文件模块接口文档`

### 13.2 type 约定

- `feat`：新增功能
- `fix`：缺陷修复
- `refactor`：重构，行为不变
- `docs`：文档更新
- `test`：测试补充或调整
- `chore`：杂项维护
- `perf`：性能优化

### 13.3 scope 约定

- `scope` 应尽量体现业务域或模块，例如：`auth`、`article`、`content`、`file`、`chat`、`follow`、`storage`、`docs`、`api`。

### 13.4 提交内容要求

- 提交标题聚焦"本次改动做了什么"，不要写成空泛描述，例如"修改代码""更新一下"。
- 一次提交只解决一类问题；不要把功能开发、重构、文档、格式整理、无关修复混在同一个提交里。
- 若改动包含接口行为变化，提交说明应体现接口或业务行为已变更。
- 提交前应先自检关键改动，至少确认代码可编译，必要时补充测试或说明未验证原因。

## 14. 什么时候提交 Git

- 完成一个独立且可验证的最小改动单元后再提交，例如一个完整功能点、一个明确 bug 修复、一次独立重构、一次文档同步更新。
- 当代码已经达到"可以回退、可以 review、不会把仓库留在半成品状态"时，应及时提交。

以下场景应单独提交：

- 新增功能完成并通过基本验证后。
- 修复 bug 并确认影响范围后。
- 完成一次独立重构且行为未变后。
- 完成接口文档、README、AGENTS、代码规范文档这类规范或文档更新后。

以下状态不应提交，除非用户明确要求保留检查点：

- 代码明显未完成。
- 项目无法编译。
- 含有临时调试代码。
- 混入与当前目标无关的改动。
