# 项目代码编写规范

本文档用于约束当前仓库的日常代码编写、重构、文档维护和 Git 提交行为。

## 1. 项目背景

- 本仓库是一个基于 Java 17、Spring Boot 4、Spring Security、MyBatis-Plus、MySQL、Redis、Knife4j/OpenAPI、MapStruct、Lombok 的博客后端项目。
- 当前核心业务模块包括 `auth`、`article`、`content`、`file`。
- 代码主目录按 `config`、`core`、`common`、`exception`、`domain`、`mapper`、`module`、`utils` 分层组织。

## 2. 分层与职责

### 2.1 Controller 层

- Controller 保持轻量，只负责参数接收、鉴权注解、调用 Service、返回统一响应。
- 不要在 Controller 中写复杂业务编排、权限分支、状态流转、计数更新、关联关系维护逻辑。
- Controller 返回统一响应包装，例如 `Result`、`PageResult`。

### 2.2 Service 层

- Service 层负责业务编排、权限校验、状态流转、事务边界、关系维护、计数同步。
- 对会影响多表数据、状态字段、统计字段的逻辑，优先在 Service 层完成统一收口。
- 同一业务动作涉及多步联动时，优先设计成可读性明确的方法，不要把关键流程拆得过碎导致理解困难。

### 2.3 Mapper / 持久化层

- 持久化查询和更新逻辑放在 MyBatis-Plus Service / Mapper 层。
- 不要把复杂业务规则塞进 Mapper 或 SQL 拼装里。
- SQL 或查询条件若牵涉业务含义，相关校验仍应放在 Service 层。

### 2.4 公共层

- 横切能力优先放在 `common`、`core`、`config`、`utils` 等公共层。
- 不要在业务模块中重复实现通用能力，例如权限判断、统一异常、序列化、存储路由、公共转换逻辑。

## 3. 实体转换规范

- 优先使用映射方式简化实体构建。
- 请求对象转实体、实体转 VO、更新已有实体、构建关联记录时，优先使用 `mapstruct`。
- 只有在 `mapstruct` 不适合的场景下，才使用 `BeanConverterUtils`，例如简单浅拷贝后再补少量运行时字段。
- 能通过 `mapstruct` 或 `BeanConverterUtils` 表达的转换，不要再手写大段 `new Entity()` + `setXxx(...)`。
- 新增映射逻辑时，优先放在对应业务模块的 convert / mapper 语义位置，避免转换代码散落在 Service 中。

## 4. 业务编写要求

### 4.1 通用要求

- 修改业务数据前，先校验归属、可见性和权限约束。
- 对新增、修改、删除流程中会影响计数、关联表、状态字段的逻辑，保持在同一个事务边界内完成联动更新。
- 对认证、文章发布、通知投递、上传任务、文件生命周期这类有状态流程，使用显式校验和明确状态流转，不要依赖隐式副作用。
- 删除业务数据时，要同步清理依赖关系、计数和临时资源。
- 除非业务明确要求，否则不要因为外部存储清理失败而阻塞本地元数据收口。

### 4.2 文件模块要求

- 文件域保持现有模式，包括上传任务跟踪、秒传复用、引用计数维护、临时分片清理。
- 上传、分片合并、秒传命中、引用回收等流程必须保证任务状态、文件状态、业务引用之间的数据一致性。

### 4.3 接口契约要求

- 新增或修改接口时，保证 Controller 路径、鉴权规则、请求模型、响应模型、Service 行为保持一致。
- 前端可见的接口契约应保持稳定；如果行为发生变化，必须同步更新接口文档。

## 5. 注释与可读性规范

### 5.1 类级注释

- 对职责不够直观的模块、服务、控制器、抽象类或接口，补充类级注释。
- 类级注释重点说明职责边界，不要写成空泛描述。

### 5.2 方法级注释

- 对不易理解的方法补充方法级 Javadoc。
- 不易理解的方法包括但不限于：
- 业务编排方法。
- 状态流转方法。
- 失败回退或补偿方法。
- 树结构组装方法。
- 访问控制与权限过滤方法。
- 解析逻辑方法。
- 关联关系重建方法。
- 计数同步方法。
- 仅看签名无法直接理解意图的私有辅助方法。
- 即使是私有 helper，只要阅读成本高，也不能省略方法文档注释。

### 5.3 注释风格

- 方法文档注释保持简洁，重点写职责、关键约束和重要副作用。
- 注释要有信息量，不要逐行复述显而易见的赋值语句。
- 只有在代码本身不够自解释时，才补充少量行内注释。

## 6. 接口文档规范

- 面向前端的接口文档位于 `docs/api文档`。
- 新增、删除、修改接口时，要同步更新对应文档，必要时更新 `docs/api文档/README.md`。
- 接口文档面向前端使用，必须写清楚：
- 请求方式。
- 请求路径。
- 鉴权要求。
- 请求字段。
- 响应字段。
- 枚举或状态含义。
- 关键边界情况和异常场景。
- 不要留下前端可见但未记录的行为变化。

## 7. 项目进度与计划文档规范

- `docs/项目进度文档.md` 用于记录当前项目真实进度、当前阶段判断、已完成能力、配套能力和风险。
- `docs/后续更新计划.md` 用于记录后续阶段目标、模块级计划、横向计划和优先级。
- 完成会影响以下信息的任务后，必须同步更新这两份文档：
- 当前已完成能力发生变化。
- 模块阶段状态发生变化。
- 当前缺口或风险判断发生变化。
- 后续优先级、阶段目标或模块计划发生变化。
- 文档更新应反映真实状态，不要机械复制提交记录，也不要把尚未完成的工作写成已完成。
- 若本次任务只涉及局部代码但不会影响项目级进度和计划判断，可以不更新这两份文档；若会影响，就不能省略。

## 8. 代码修改预期

- 优先做增量式修改，尽量保持各模块既有风格和约定。
- 修改旧代码时，如果附近代码的可读性问题会直接影响本次改动理解，可以顺手做小范围整理。
- 不要把无关重构、格式清理、行为修改混在一次任务里，除非它们强相关且必须同时完成。
- 对非平凡代码改动，除非环境阻塞，否则至少执行一次 `mvn -q -DskipTests compile` 进行验证。

## 9. Git 提交规范

### 9.1 提交信息格式

- 提交信息使用清晰、可检索的中文或中英混合风格。
- 推荐格式：`type(scope): 简要说明`

示例：

- `feat(file): 新增分片上传完成接口`
- `fix(auth): 修复刷新令牌校验失败问题`
- `refactor(article): 收敛文章保存流程的分类标签同步逻辑`
- `docs(api): 更新文件模块接口文档`

### 9.2 type 约定

- `feat`：新增功能
- `fix`：缺陷修复
- `refactor`：重构，行为不变
- `docs`：文档更新
- `test`：测试补充或调整
- `chore`：杂项维护
- `perf`：性能优化

### 9.3 scope 约定

- `scope` 应尽量体现业务域或模块，例如：`auth`、`article`、`content`、`file`、`storage`、`docs`、`api`。

### 9.4 提交内容要求

- 提交标题聚焦“本次改动做了什么”，不要写成空泛描述，例如“修改代码”“更新一下”。
- 一次提交只解决一类问题；不要把功能开发、重构、文档、格式整理、无关修复混在同一个提交里。
- 若改动包含接口行为变化，提交说明应体现接口或业务行为已变更。
- 提交前应先自检关键改动，至少确认代码可编译，必要时补充测试或说明未验证原因。

## 10. 什么时候提交 Git

- 完成一个独立且可验证的最小改动单元后再提交，例如一个完整功能点、一个明确 bug 修复、一次独立重构、一次文档同步更新。
- 当代码已经达到“可以回退、可以 review、不会把仓库留在半成品状态”时，应及时提交。

以下场景应单独提交：

- 新增功能完成并通过基本验证后。
- 修复 bug 并确认影响范围后。
- 完成一次独立重构且行为未变后。
- 完成接口文档、README、AGENTS、代码规范文档这类规范或文档更新后。

以下状态不应提交，除非用户明确要求保留检查点：

- 代码明显未完成。
- 项目无法编译。
- 含有临时调试代码。
- 混入与当前目标无关的改动。
