# 项目结构规范

本文档只约束“代码应该放在哪里、目录如何组织、文件如何命名”。编码风格、业务实现、异常、日志、缓存、测试、接口文档和 Git 提交规则不在本文档展开，统一以 [项目代码编写规范](./项目代码编写规范.md) 为准。

## 1. 适用边界

本文档关注：

- 顶层包目录职责。
- 业务模块目录结构。
- 子域化组织方式。
- 模型、转换器、Repository、Controller、Service 等文件落位。
- 新增模块、新增接口、新增公共能力时的目录选择。

本文档不关注：

- Controller、Service、Repository 内部应该如何写业务逻辑。
- 事务、异常、日志、缓存和测试应该如何实现。
- Git 提交和编译验证要求。

以上编码问题统一查看 [项目代码编写规范](./项目代码编写规范.md)。

## 2. 结构总原则

- 项目保持“公共基础层 + 领域实体层 + 按业务域拆分的模块层”结构。
- 业务能力优先落在 `module/<domain>` 下。
- 公共层只放横切能力、基础设施能力、通用工具和全局配置。
- 新增目录必须有明确职责，不为对齐模板空建目录。
- 新增代码优先接入现有目录；只有职责稳定且边界清晰时才新增目录层级。

## 3. 顶层包结构

主包目录固定为：

```text
src/main/java/com/cybzacg/blogbackend
├─ config
├─ common
├─ core
│  └─ util
├─ domain
│  ├─ ai / article / auth / chat / config / content
│  ├─ dashboard / file / follow / forum / notice / report / system
│  └─ (按业务域分包)
├─ enums
├─ exception
├─ mapper
│  ├─ ai / article / auth / chat / config / content
│  ├─ dashboard / file / follow / forum / notice / report / system
│  └─ (按业务域分包，与 domain 对齐)
├─ module
└─ utils
```

## 4. 顶层目录职责

### 4.1 `config`

- 放项目级 Spring 配置、中间件配置、属性绑定、线程池、缓存、WebSocket、存储、邮件、安全等配置。
- 模块私有配置优先放入 `module/<domain>/config`。

### 4.2 `common`

- 放跨模块复用的常量、Redis 工具、存储抽象、切面、注解等能力。
- 单一业务模块专属内容不放入 `common`。

### 4.3 `core`

- 放统一 Web 响应、过滤器、安全校验、请求上下文等运行时核心链路能力。
- `core/util` 放仅被核心链路使用的工具类。

### 4.4 `domain`

- 放数据库实体。
- 按业务域分包，例如 `domain/article`、`domain/chat`、`domain/report`。
- 不放 Request、VO、PageQuery、展示聚合对象。

### 4.5 `enums`

- 放枚举、结果码、状态码。
- 模块专属枚举优先放在 `enums/<domain>`。

### 4.6 `exception`

- 放异常定义和统一异常处理器。

### 4.7 `mapper`

- 放 MyBatis-Plus Mapper 接口。
- 分包与 `domain` 保持业务域对齐。
- XML 文件放入 `src/main/resources/.../mapper/<domain>/`。

### 4.8 `module`

- 放全部业务域实现，是业务代码默认落点。
- 一级子目录表示业务域，例如 `auth`、`article`、`content`、`file`、`chat`、`follow`、`ai`、`report`、`dashboard`、`forum`。

### 4.9 `utils`

- 放纯工具性质、无业务状态、无模块归属、无 Spring 业务编排依赖的通用工具。

## 5. 业务模块标准结构

简单模块默认结构：

```text
module/<domain>
├─ controller
├─ service
│  └─ impl
├─ model
├─ convert
├─ repository
│  └─ impl
├─ config
├─ constant
├─ task
└─ websocket
```

不是所有模块都必须拥有全部子目录。没有实际文件时不创建空目录。

### 5.1 标准子目录职责

- `controller`：接口入口。
- `service`：业务接口。
- `service/impl`：业务实现。
- `model`：请求对象、查询对象、VO、内部传输模型。
- `convert`：MapStruct 映射器。
- `repository`：模块内部数据访问抽象。
- `repository/impl`：Repository 实现。
- `config`：模块私有配置。
- `constant`：模块私有常量。
- `task`：模块专属调度任务、异步收口任务。
- `websocket`：WebSocket 或实时协议处理。

## 6. 复杂模块子域化结构

当模块文件数较多或职责边界清晰时，按子域组织。每个子域拥有自己的标准分层。

当前子域化模块如下。

### 6.1 `auth`

```text
module/auth
├─ account
├─ audit
├─ author
├─ config
├─ experience
├─ notice
└─ rbac
```

- `account`：认证、令牌、用户管理、MFA。
- `audit`：审计日志、操作日志。
- `author`：作者申请、作者主页、作者权限。
- `config`：系统配置管理。
- `experience`：经验值、等级。
- `notice`：通知中心、通知投递、通知偏好。
- `rbac`：角色权限、菜单管理、用户角色分配。

### 6.2 `chat`

```text
module/chat
├─ shared
├─ conversation
├─ member
├─ message
├─ attachment
├─ governance
├─ push
└─ websocket
```

- `shared`：聊天模块内部共享常量、模型、转换器、辅助类。
- `conversation`：会话编排、大厅、话题频道。
- `member`：成员管理、加入申请、邀请链接。
- `message`：消息发送、编辑、删除、已读回执。
- `attachment`：附件处理、图片和语音处理。
- `governance`：管理员治理、审核、指标统计。
- `push`：消息推送、通知、Redis 订阅。
- `websocket`：连接、协议编解码。

### 6.3 `content`

```text
module/content
├─ collection
├─ comment
├─ footprint
├─ interaction
├─ shared
└─ taxonomy
```

- `collection`：收藏、收藏夹。
- `comment`：评论。
- `footprint`：浏览足迹。
- `interaction`：点赞、互动。
- `shared`：内容模块内部共享转换器和服务。
- `taxonomy`：分类、标签。

### 6.4 子域化落位规则

- 新增代码优先落入对应子域。
- 子域内按标准结构组织，如 `service`、`model`、`repository`。
- 跨子域共享内容放入当前模块的 `shared`。
- 不把子域专属文件放回模块根层。

## 7. `model` 目录结构

模块模型统一放在 `model` 下，并按使用场景继续分层：

- `model/admin`：后台接口请求与响应模型。
- `model/user`：登录用户侧接口请求与响应模型。
- `model/publics`：匿名公开接口模型。当前仓库使用 `publics`，不新增 `public` 平行目录。
- `model/common`：当前模块多个接口共用的模型。
- `model/data`：查询结果承载对象、聚合数据项、中间结果模型。
- `model/internal`：仅模块内部使用的内部协议模型。
- `model/websocket`：实时消息协议模型。

子域化模块中，`model` 放在对应子域内部，例如 `module/chat/message/model/user/`。

## 8. 命名规则

### 8.1 Controller 命名

- 用户侧接口：`UserXxxController`。
- 公开匿名接口：`PublicXxxController`。
- 后台接口：`XxxAdminController` 或 `SysXxxAdminController`。

### 8.2 Service 命名

- 业务接口：`XxxService`。
- 业务实现：`XxxServiceImpl`。
- 后台业务：`XxxAdminService`、`XxxAdminServiceImpl`。
- 公开查询：`PublicXxxService`、`PublicXxxServiceImpl`。

### 8.3 Model 命名

- 请求对象：`XxxRequest`。
- 分页筛选：`XxxPageQuery`。
- 普通筛选：`XxxQuery`。
- 前端响应：`XxxVO`。
- 聚合项 / 内部载体：`XxxItem`、`XxxPayload`、`XxxInfo`、`XxxEnvelope`。
- 不使用含义模糊的 `DTO` 泛称。

### 8.4 Convert 命名

- 映射器使用 `XxxModelConvert` 或既有模块约定名称。
- 子域共享映射器放入 `shared/convert`。

### 8.5 Repository 命名

- 接口：`XxxRepository`。
- 实现：`XxxRepositoryImpl`。

## 9. 新增功能落位流程

### 9.1 新增接口

新增接口时检查以下文件落位：

- `controller`：接口入口。
- `model`：请求、查询、响应模型。
- `service` / `service/impl`：业务能力。
- `convert`：实体和模型转换。
- `repository` / `repository/impl`：复杂查询或数据访问收口。
- `docs/api文档`：接口文档。

### 9.2 新增业务模块

新增独立业务模块时：

1. 在 `module` 下新增业务域目录。
2. 建立必要的 `controller`、`service`、`model`。
3. 按需建立 `convert`、`repository`、`constant`、`config`、`task`。
4. 若有数据库实体，同步新增 `domain/<domain>`、`mapper/<domain>` 和 XML 目录。
5. 更新本文档中对应模块说明。

### 9.3 新增公共能力

新增公共能力时优先选择：

- 项目级配置：`config`。
- 运行时核心链路：`core`。
- 多模块复用能力：`common`。
- 无状态纯工具：`utils`。

若只服务单个模块，应留在对应 `module/<domain>` 内部。

## 10. 测试与文档结构

### 10.1 测试目录

- `src/test/java/com/cybzacg/blogbackend` 下测试包路径默认与生产模块路径对齐。
- 模块测试优先按 `module/<domain>` 组织。
- 测试类名应能直接对应目标类或目标链路。

### 10.2 文档目录

- 项目入口：`docs/README.md`。
- API 文档：`docs/api文档`。
- 阶段任务：`docs/tasks`。
- 需求文档：`docs/需求文档`。
- 前端联调：`docs/前端`。

## 11. 禁止的结构做法

- 禁止把业务专属逻辑放入 `common`、`core`、`utils`。
- 禁止把 Request、VO、PageQuery 放入 `domain`。
- 禁止把工具类、枚举、服务辅助类塞进 `model`。
- 禁止新增 `temp`、`misc`、`other` 等含义不清的目录。
- 禁止同时存在 `public` 和 `publics` 两套公开模型目录。
- 禁止新增与既有业务域平行但语义重复的模块。
- 禁止为了单个接口新增一级业务模块。
- 禁止创建空目录只为满足模板。

## 12. 结构变更同步要求

以下变化必须更新本文档或相关导航：

- 新增或删除业务模块。
- 顶层目录职责变化。
- 模块标准结构变化。
- 子域化结构变化。
- 模型目录规则变化。
- 文档入口变化。

只涉及编码实现、测试、异常、日志、缓存、Git 等规则时，更新 [项目代码编写规范](./项目代码编写规范.md)，不要写入本文档。
