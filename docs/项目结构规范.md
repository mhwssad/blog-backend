# 项目结构规范

本文档用于约束当前仓库的目录分层、模块组织、模型落位、命名方式和新增代码的放置规则。

本文档回答两个问题：

- 代码应该放在哪里。
- 新增模块或新增功能时，应该按什么结构组织。

如无特殊说明，当前仓库的结构调整、模块新增、业务扩展、重构拆分，都必须同时遵守本文档与 [项目代码编写规范](./项目代码编写规范.md)。

## 1. 适用范围

- 适用于 `src/main/java/com/cybzacg/blogbackend` 下的全部生产代码。
- 适用于 `src/test/java/com/cybzacg/blogbackend` 下按模块对齐的测试代码组织。
- 适用于 `docs` 下与项目结构、接口、进度、计划相关的文档同步。

## 2. 结构总原则

- 项目保持“公共基础层 + 领域实体层 + 按业务域拆分的模块层”结构，不使用按技术动作随意堆目录的方式组织业务代码。
- 业务能力必须优先落在 `module/<业务域>` 下，不要把具体业务规则散落到 `common`、`utils` 或 `config`。
- 公共层只沉淀横切能力、基础设施能力、通用工具和全局配置，不承载单一业务域的特例逻辑。
- 新增代码优先增量接入现有目录；只有在职责明确、可复用、能形成稳定边界时，才允许新增目录层级。
- 依赖方向必须保持单向清晰，禁止跨层穿透和反向依赖。

## 3. 顶层包结构约束

当前主包目录固定为：

```text
src/main/java/com/cybzacg/blogbackend
├─ config
├─ common
├─ core
├─ domain
├─ enums
├─ exception
├─ mapper
├─ module
└─ utils
```

各目录职责如下。

### 3.1 `config`

- 放 Spring 配置、中间件配置、属性绑定、线程池、缓存、WebSocket、存储、邮件、安全等基础配置。
- 只放“项目级配置”，不放具体业务编排。
- 业务模块专属、且不会被其他模块复用的轻量配置，可放在对应 `module/<domain>/config`。

### 3.2 `common`

- 放横切常量、Redis 工具、存储抽象、切面、注解等跨模块复用能力。
- 进入 `common` 的前提是“至少可被多个业务域稳定复用”，而不是“当前写着方便”。
- 禁止把某个单一模块的条件分支、状态判断、查询拼装塞进 `common`。

### 3.3 `core`

- 放统一 Web 响应、过滤器、安全校验、请求上下文等运行时核心链路能力。
- `core` 面向全项目运行链路，不面向单一业务域。
- 禁止把普通业务 Service 或模块模型放入 `core`。

### 3.4 `domain`

- 放数据库实体，对应表结构和持久化对象。
- `domain` 只表达持久化数据，不承担接口请求、前端响应或复杂展示聚合职责。
- 禁止将 `VO`、`Request`、页面查询对象混放进 `domain`。

### 3.5 `enums`

- 放跨模块或明确领域内的枚举、结果码、状态码。
- 若某类枚举只服务某个模块，优先放在 `enums/<domain>`，而不是继续塞进模块 `model`。

### 3.6 `exception`

- 放业务异常定义与统一异常处理器。
- 模块级异常优先复用通用 `BusinessException` + 错误码表达；只有异常语义非常稳定且确有必要时才新增专用异常类型。

### 3.7 `mapper`

- 放 MyBatis-Plus Mapper 接口，与 `src/main/resources/.../mapper/*.xml` 对应。
- `mapper` 只承载数据访问接口，不承载业务编排。
- 通用持久化查询可以下沉到模块 `repository`，但基础 Mapper 入口仍统一放在顶层 `mapper`。

### 3.8 `module`

- 放全部业务域实现，是主要业务代码入口。
- 每个一级子目录表示一个独立业务域，例如 `auth`、`article`、`content`、`file`、`chat`、`follow`。
- 新增业务能力时，优先判断是扩展现有域，还是确实需要新增独立域；禁止因单个接口而随意拆新模块。

### 3.9 `utils`

- 放纯工具性质、无业务状态、无模块归属、无 Spring 业务编排依赖的通用工具。
- 只有真正横向复用、职责清晰的辅助能力才进入 `utils`。
- 禁止把“暂时没想好放哪”的代码丢进 `utils`。

## 4. 业务模块标准结构

`module/<domain>` 是业务开发的默认落点。模块下优先按以下结构组织：

```text
module/<domain>
├─ controller
├─ service
│  └─ impl
├─ model
├─ convert
├─ repository
│  └─ impl
├─ config
├─ constant
├─ task
└─ websocket
```

其中：

- `controller`：仅放接口入口，负责参数接收、鉴权注解、调用 Service、返回统一响应。
- `service`：放业务接口。
- `service/impl`：放业务编排实现、事务收口、权限校验、状态流转、关系维护。
- `model`：放请求对象、查询对象、VO、内部传输模型。
- `convert`：放 MapStruct 映射器，负责实体与模型之间的转换。
- `repository`：放模块内部的数据访问抽象，承接复杂查询、查询条件组装、分页读取、局部读写收口。
- `repository/impl`：放 Repository 实现。
- `config`：放模块私有配置，例如聊天域专属 Redis 推送配置。
- `constant`：放模块私有常量。
- `task`：放模块专属调度任务、异步收口任务。
- `websocket`：仅用于 WebSocket 或实时协议处理，不得混入普通 HTTP Service。

不是所有模块都必须拥有全部子目录，但新增目录必须有明确职责，不能为对齐模板而空建。

## 5. 模块内细分规则

### 5.1 `controller`

- 控制器按使用对象拆分，如用户侧、后台侧、公开侧。
- 命名优先体现边界，例如 `UserXxxController`、`PublicXxxController`、`XxxAdminController`。
- 禁止在 Controller 中直接注入多个 Mapper 拼业务。

### 5.2 `service` 与 `service/impl`

- `service` 接口表达业务能力边界，不表达底层 SQL 细节。
- `service/impl` 内允许组合多个 Repository、Mapper、公共服务，但必须保持单一业务动作的可读性。
- 会影响多表、状态、计数、关联关系的逻辑，统一收口在 `service/impl` 的事务边界内。

### 5.3 `model`

模块模型统一放在 `model` 下，并按使用场景继续分层：

- `model/admin`：后台接口请求与响应模型。
- `model/user`：登录用户侧接口请求与响应模型。
- `model/publics`：匿名公开接口模型。当前仓库已使用 `publics` 目录，后续保持一致，不再新增 `public` 平行目录。
- `model/common`：当前模块内多个接口共用、但不适合放到公共层的共享模型。
- `model/data`：模块内部查询结果承载对象、聚合数据项、中间结果模型。
- `model/internal`：仅模块内部使用、不会直接暴露给前端或其他模块的内部协议模型。
- `model/websocket`：仅实时消息协议模型，限聊天等实时模块使用。

命名规则：

- 请求对象使用 `Request` 结尾，例如 `ArticleSaveRequest`。
- 分页或筛选条件使用 `PageQuery` 或明确语义的 `Query` 结尾。
- 返回前端展示对象使用 `VO` 结尾。
- 中间聚合数据项使用 `Item`、`Payload`、`Info`、`Envelope` 等能够表达语义的名称，不使用含糊的 `DTO` 泛称。

禁止项：

- 禁止把数据库实体复制一份改名为 `VO` 后直接当接口模型。
- 禁止把管理端、用户端、公开端字段差异明显的对象强行合并成一个“万能模型”。
- 禁止在 `model` 下继续堆放与模型无关的工具类、枚举、服务辅助类。

### 5.4 `convert`

- 实体转 VO、Request 转实体、更新已有实体、关系记录构建，优先放在 `convert` 下统一处理。
- 模块内映射器命名使用 `XxxModelMapper`。
- 若模块复杂度较高，可按业务子域拆多个映射器，但都应保持在 `convert` 下管理。

### 5.5 `repository`

- 当模块存在复杂查询、分页拼装、可见性过滤、聚合读取，或需要逐步清理 Service 直连 Mapper 时，应优先新增 `repository` 收口。
- Repository 负责“如何查、如何写”的数据访问组织，不负责“为什么这么做”的业务决策。
- Service 可以依赖 Repository，Repository 可以依赖 Mapper；禁止反向依赖。

### 5.6 可选目录

- `authentication`：仅认证域内用于认证流程组件，例如认证 Provider、认证 Token 处理器。
- `token`：仅认证域内用于令牌管理抽象与实现。
- `task`：只放与当前模块强绑定的调度任务、恢复任务、异步处理任务。
- `websocket`：只放消息编解码、握手后处理、连接处理器等实时通道代码。

新增可选目录前，必须先判断是否已经有现成目录能承载，不得为了“分类看起来更细”而随意增加层级。

## 6. 依赖方向与调用边界

默认依赖方向如下：

```text
controller -> service -> repository -> mapper -> database
controller -> service -> mapper -> database
service -> convert
service -> domain / model
repository -> domain
```

必须遵守以下规则：

- Controller 不直接依赖 Repository。
- Controller 不直接依赖 Mapper。
- Repository 不调用 Service。
- `common`、`core`、`utils` 不反向依赖具体业务模块。
- 一个模块可以复用另一个模块的稳定服务能力，但禁止跨模块直接复制实现或绕开边界读取对方私有细节。
- 涉及跨模块联动时，优先通过 Service 能力收口，不要在本模块 Service 中直接拼装对方多张表的查询逻辑。

## 7. 新增功能的默认落位规则

### 7.1 新增接口

新增一个接口时，默认需要同时判断以下落点：

- `controller`：新增接口入口。
- `model`：新增请求对象、查询对象、响应对象。
- `service`：新增或扩展业务能力。
- `convert`：若存在实体和模型转换，补充映射方法。
- `repository`：若查询条件复杂、聚合读取重，优先新增或扩展 Repository。
- `docs/api文档`：同步更新前端可见接口文档。

### 7.2 新增业务模块

新增独立业务模块时，默认步骤如下：

1. 在 `module` 下新增一级业务目录。
2. 至少建立 `controller`、`service`、`model` 三层。
3. 若存在明确的实体模型转换，建立 `convert`。
4. 若存在复杂查询或计划逐步去 Mapper 直连，建立 `repository`。
5. 若模块存在私有常量、配置、调度或协议处理，再按需新增 `constant`、`config`、`task`、`websocket`。

### 7.3 新增公共能力

新增公共能力前，必须先满足以下条件之一：

- 明确会被多个模块复用。
- 明确属于项目级运行时基础设施。
- 不放到公共层会导致多个模块重复维护同一套逻辑。

不满足上述条件时，优先留在业务模块内部。

### 7.4 预备模块

部分模块处于"预备"状态，即骨架结构已建立、核心能力已落地，但仍有规划中的子能力尚未实现。预备模块遵循与正式模块相同的目录结构和编码规范，区别仅在于：

- 新增代码仍按标准结构落入对应子包，不做特殊处理。
- 未实现的子能力在对应任务清单中标注 ⏳，不阻塞其他模块对该模块已有接口的依赖。
- 预备模块升级为正式模块后，移除任务清单中的 ⏳ 标记即可。

当前预备模块：`ai`（同步聊天已落地，Agent 异步任务待实现）、`report`（举报 CRUD 已落地，高级治理联动待扩展）。

## 8. 测试与文档的结构同步规则

### 8.1 测试结构

- `src/test/java/com/cybzacg/blogbackend` 下的测试包路径，默认与生产代码模块路径保持同域对齐。
- 模块测试优先按 `module/<domain>` 组织，避免测试代码重新按技术类型拆散。
- 控制器、Service、Repository、工具类测试命名要能直接对应目标类或目标链路。

### 8.2 文档同步

以下结构变化必须同步更新文档：

- 新增或删除业务模块。
- 顶层目录职责发生变化。
- 模块标准结构发生变化。
- 接口模型组织方式发生变化。
- 文档入口发生变化。

同步范围至少包括：

- `docs/README.md`
- `docs/项目代码编写规范.md` 或本文档
- `docs/tasks/README.md` 或对应任务清单
- `docs/api文档` 中受影响的接口文档

## 9. 强制禁止项

- 禁止在 Controller 中编排业务、维护状态流转、更新计数或拼装多表逻辑。
- 禁止把某个模块的专属逻辑塞进 `common`、`core` 或 `utils` 伪装成“通用能力”。
- 禁止在 Service 中长期散落大段手写实体转换，而不收口到 `convert`。
- 禁止把请求对象、响应对象、数据库实体混放为同一类对象复用全部场景。
- 禁止模块内直接复制其他模块的模型、常量或查询逻辑。
- 禁止为了临时需求新增含义不清的目录，例如 `temp`、`misc`、`helper`、`other`。
- 禁止把“暂时没想好放在哪”的代码直接丢进 `utils`。
- 禁止在没有明确职责提升的情况下增加目录层级，导致结构比业务本身更难理解。

## 10. 落地要求

- 后续代码开发、重构、目录调整、模块扩展，都以本文档作为项目结构口径。
- Review 时除检查行为正确性外，还必须检查代码落位是否符合本文档。
- 若现有历史代码与本文档不完全一致，后续改动时优先做增量收口，不要求一次性大规模搬迁。
- 当现有结构无法覆盖新场景时，应先更新本文档，再扩展目录，而不是先写代码后补规则。
