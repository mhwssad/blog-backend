# AGENTS 强制规范入口

当前项目的代码开发、重构、接口调整、注释补充、文档更新和 Git 提交，必须遵循本文件以及 `docs` 下的规范文档。

## 一、规范职责边界

- [项目代码编写规范](docs/项目代码编写规范.md) 是代码实现行为的主文档，只约束“怎么写、怎么验证、怎么提交”。
- [项目结构规范](docs/项目结构规范.md) 是代码落位和目录组织的主文档，只约束“放哪里、目录怎么组织、文件怎么命名”。
- [任务清单导航](docs/tasks/README.md) 是当前阶段任务入口。当前项目任务已切换到第二期规划，不再按已删除的一期任务清单推进。
- 如无明确例外，所有代码改动必须同时满足本文件、代码编写规范和结构规范。
- 当零散说明与上述规范冲突时，优先级为：本文件 > `docs/项目代码编写规范.md` / `docs/项目结构规范.md` > 其他说明。

## 二、代码编写强约束

以下规则在 [项目代码编写规范](docs/项目代码编写规范.md) 中有完整定义，此处只列强约束摘要。

### 2.1 分层编码职责

- Controller 只做参数接收、参数校验注解、鉴权注解、调用 Service 和统一响应包装。
- Controller 不直接操作 Mapper、Repository 或数据库。
- Service 负责业务编排、权限校验、状态流转、事务边界、关系维护和计数同步。
- Repository 负责数据访问组织，不承担业务决策，不调用 Service。
- Mapper 和 XML 只承载数据访问，不写业务状态流转和权限判断。
- 已收口到 Repository 的查询条件，禁止在 Service 中绕过重新拼装。

### 2.2 实体转换

- 请求对象转实体、实体转 VO、更新已有实体、构建关联记录时，优先使用 MapStruct。
- 仅在 MapStruct 明显不适合时使用 `BeanConverterUtils`。
- 禁止在可映射场景中手写大段 `new Entity()` + `setXxx(...)`。
- 转换逻辑不得散落在 Service 主流程中。

### 2.3 业务编写

- 新增、修改、删除业务数据前，必须完成归属、权限、可见性和状态合法性校验。
- 多表联动、状态流转、计数更新必须收口在统一事务边界内。
- 文件、聊天、治理与安全等高一致性模块，必须保证任务状态、业务状态、关联引用、审计记录之间的数据一致性。
- 举报处理、封禁、禁言、内容隐藏等治理动作必须调用对应业务服务，不直接绕过一致性逻辑改表。
- 当前处于开发阶段时，数据库结构调整默认直接修改原始建表脚本；只有用户明确要求时，才单独补迁移脚本。

### 2.4 异常处理

- 统一使用 `BusinessException` + `ResultCode` / `ResultErrorCode` 表达业务异常。
- Service 优先通过 `ExceptionThrowerCore` 抛出业务异常。
- 不要在 Service 中手动捕获业务异常后重新包装。
- 不要吞异常；补偿完成后应抛出原始异常或更具语义的异常。

### 2.5 注释与可读性

- 对职责不直观的服务、控制器、抽象类、接口、策略类补充类级注释。
- 对业务编排、状态流转、失败补偿、权限过滤、解析逻辑、关联关系重建、计数同步等不易理解的方法补充方法级 Javadoc。
- 私有 helper 只要阅读成本高，也必须补充方法文档注释。
- 注释写职责、关键约束和重要副作用，不逐行复述显而易见的代码。

### 2.6 测试

- 新增或修改 Service 方法时，必须补充对应服务级测试。
- 测试应覆盖正常路径、边界条件和异常分支。
- 权限、归属、状态流转、计数和事务联动属于高风险路径，必须优先覆盖。
- 若用户明确要求暂缓测试，必须在对应任务文档中标明测试补强项。

### 2.7 接口文档

- 新增、删除、修改接口时，必须同步更新 `docs/api文档`。
- 前端可见行为变化不得只改代码不改文档。
- 接口文档必须写清楚请求方式、路径、鉴权、请求字段、响应字段、枚举含义、异常场景和关键边界。

### 2.8 项目任务文档

- 新增、调整或完成具体研发任务时，必须同步更新 `docs/tasks` 下对应任务清单。
- 影响项目阶段、已完成能力、缺口风险或后续路线时，必须同步更新 `docs/tasks/README.md`。
- 当前二期任务入口为 [docs/tasks/README.md](docs/tasks/README.md)。

### 2.9 编译验证

- 非平凡代码改动在环境允许时，必须至少执行一次 `mvn -q -DskipTests compile`。
- 文档-only 改动不强制编译，但提交前应检查链接和引用是否仍然有效。

## 三、结构落位强约束

以下规则在 [项目结构规范](docs/项目结构规范.md) 中有完整定义，此处只列强约束摘要。

- 顶层包目录、业务模块目录、子域化结构和模型命名，以结构规范为准。
- 业务能力优先落在 `module/<domain>` 下，不把业务专属逻辑放入 `common`、`core`、`utils`。
- `domain` 只放数据库实体；禁止放 Request、VO、PageQuery、展示聚合对象。
- `model` 下按 `admin`、`user`、`publics`、`common`、`data`、`internal`、`websocket` 等场景分层。
- 当前公开模型目录使用 `publics`，禁止新增平行 `public` 目录。
- 子域化模块新增代码优先落入对应子域，跨子域共享内容放入当前模块的 `shared`。
- 禁止新增 `temp`、`misc`、`other` 等含义不清的目录。
- 禁止为了单个接口新增一级业务模块。
- 禁止创建空目录只为满足模板。

## 四、Git 强制要求

- 提交信息应使用清晰、可检索的格式：`type(scope): 简要说明`。
- 一次提交只能解决一类问题，禁止把功能开发、重构、文档更新、格式整理和无关修复混在同一个提交里。
- 完成一个独立、可验证、可 review 的最小改动单元后，才应提交 Git。
- 禁止在明显未完成、无法编译、含临时调试代码、混入无关改动的状态下提交，除非用户明确要求保留检查点。
- 分步提交：每完成一个逻辑步骤后立即提交，保持提交粒度小且可追溯。
- 工作区存在他人或用户已有改动时，不得擅自回滚、覆盖或混入提交；只提交本次任务相关文件。
