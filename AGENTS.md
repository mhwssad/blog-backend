# AGENTS 强制规范入口

当前项目的代码开发、重构、接口调整、注释补充、文档更新、提交 Git，必须遵循以下规则。

## 一、总准则

- `docs/项目代码编写规范.md` 是当前仓库代码编写要求的主文档，默认以该文档为准。
- 如无明确例外，所有代码改动都必须同时满足本文件与 `docs/项目代码编写规范.md` 的要求。
- 当本文件与其他零散说明冲突时，优先以本文件和 `docs/项目代码编写规范.md` 为准。

## 二、强制规范

- 必须保持分层清晰：Controller 只做参数接收、鉴权注解、调用 Service、统一响应包装；业务编排必须放在 Service 层。
- 必须优先使用 `mapstruct` 做实体转换；仅当 `mapstruct` 明显不适合时，才允许使用 `BeanConverterUtils`。
- 禁止在可通过 `mapstruct` 或 `BeanConverterUtils` 完成的场景中，继续手写大段 `new Entity()` + `setXxx(...)`。
- 必须在修改业务数据前完成归属、权限、可见性、状态合法性校验。
- 对涉及多表联动、状态流转、计数更新、关联关系维护的逻辑，必须在统一事务边界内收口。
- 对不易理解的方法，必须补充方法级 Javadoc；私有 helper 只要阅读成本高，也不能省略文档注释。
- 新增、删除、修改接口时，必须同步更新 `docs/api文档` 中对应文档；前端可见行为变化不得只改代码不改文档。
- 完成会影响项目当前进度、已完成能力、阶段结论、后续优先级或模块计划的任务后，必须同步更新 `docs/项目进度文档.md` 与 `docs/后续更新计划.md`，不得只改代码或只改局部文档而不维护这两份项目级文档。
- 文件模块代码必须保持上传任务、秒传复用、引用计数、临时分片清理等关键流程的一致性，不得破坏现有收口模式。
- 非平凡代码改动在环境允许时，必须至少执行一次 `mvn -q -DskipTests compile` 验证。

## 三、Git 强制要求

- 提交信息应使用清晰、可检索的格式，推荐：`type(scope): 简要说明`。
- 一次提交只能解决一类问题，禁止把功能开发、重构、文档更新、无关修复混在同一个提交里。
- 完成一个独立、可验证、可 review 的最小改动单元后，才应提交 Git。
- 禁止在明显未完成、无法编译、含临时调试代码、混入无关改动的状态下提交，除非用户明确要求保留检查点。
